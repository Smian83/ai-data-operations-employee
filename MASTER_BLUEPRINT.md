## 1. Executive Summary

AI Data Operations Employee is a multi-tenant SaaS backend that performs data operations work — ingesting, profiling, cleaning, standardizing, matching, deduplicating, and exporting a customer's data — automatically, safely, and under explicit human control. The system is built as two independently deployable processes (a FastAPI HTTP API and a background worker pool) sharing one PostgreSQL database, with every domain table structurally scoped to a tenant (`organization_id`) from the schema up.

As of this document's writing, twelve modules are complete and merged to `main`: foundation and health checking (Module 1), multi-tenant authentication (Module 2), the core domain model of data sources and tasks (Module 3), a lease-fenced task execution engine (Module 4), CSV ingestion and profiling (Module 5, with a security-hardening follow-up, 5.1), deterministic data cleaning with human approval (Module 6), field-level data standardization (Module 7), deterministic record matching and deduplication (Module 8), deduplicated data export (Module 9), secure artifact retrieval (Module 10), an operational review queue (Module 11), and scheduled recurring task execution (Module 12).

The system deliberately contains no AI or machine-learning component anywhere in its execution path. Every transformation — cleaning, standardization, matching — is a deterministic, versioned, pure-function rule engine whose output is fully reproducible and independently auditable. "AI Data Operations Employee" describes the product's role (performing data-operations labor a human employee would otherwise do manually) and its future multi-agent shape (Section 14), not a claim that machine learning drives any decision made today.

The architecture that emerged across these twelve modules is consistent and intentional, not accidental: every domain table is tenant-scoped by a composite foreign key; every output-producing pipeline stage requires an approved upstream stage before it will run; every approval-gated action follows the same `pending_review → approved/rejected → optionally rolled_back` state machine; every audit table follows the same "exact aggregate, capped-and-deterministic detail" persistence pattern; and every new task type is added by writing one new handler and registering it, never by modifying the execution engine itself. This document names these patterns explicitly so that Module 13 and beyond extend them rather than reinvent them.

## 2. Vision

An organization should be able to hand off the repetitive, error-prone parts of managing their operational data — reconciling messy CSV exports, normalizing inconsistent formats, finding and merging duplicate records, producing a clean deliverable — to a system that performs the work reliably, on schedule, and transparently, while keeping a human in control of every consequential decision. The system should behave the way a careful, methodical employee would: it does the mechanical work completely and consistently every time, explains what it did and why, and stops to ask before doing anything that can't be undone.

This vision is scoped deliberately to data operations, not general business process automation. The system's competence is in taking data from an unreliable source state to a reliable, exportable state through a well-defined pipeline of stages, each independently reviewable. It is not a general-purpose workflow engine, not a business-intelligence or analytics product, and not an autonomous decision-maker over business outcomes — see Section 7 (Non-Goals) for the explicit boundary.

## 3. Mission

Build a production-grade, multi-tenant backend that executes a deterministic, auditable data-operations pipeline — sync, clean, standardize, match, export — on a customer's data, gated at every consequential step by explicit human approval, and do so with the same rigor a regulated financial or healthcare backend would require: structural tenant isolation, encrypted secrets, append-only audit trails, idempotent and retry-safe execution, and migrations and behavior verified against real PostgreSQL before every release, not just a fast local sandbox.

Each module in this project's history has shipped only after an explicit design → review → approval → implementation → verification → report → approval → merge cycle (see Section 24 and Section 26). The mission is not just "build the pipeline" but "build it the same disciplined way, every time," because the discipline is what makes the resulting system trustworthy enough for a customer to route their real operational data through it.

## 4. Guiding Engineering Principles

These principles were not written in advance and then followed; they emerged from twelve modules of real decisions and were later recognized as consistent patterns worth stating explicitly. Every principle below is backed by at least one concrete precedent in the codebase.

**Tenant isolation is structural, not incidental.** Every domain table carries `organization_id`. Every tenant-scoped foreign key uses the composite pattern `(organization_id, X) → parent(organization_id, id)`, which requires an explicit `UniqueConstraint(organization_id, id)` on the referenced parent — this is enforced by the database, not just application code, so a bug in a query's `WHERE` clause cannot leak cross-tenant data through a foreign key join. Cross-organization access is uniformly a `404`, indistinguishable from "does not exist," never a `403` that would confirm a resource's existence to an unauthorized caller. File-based tenant isolation (CSV input, cleaned, standardized, and exported roots) uses the same structural approach: a path is never resolved against a shared root, only against that tenant's own subdirectory, with traversal-escape checks layered on top, not instead.

**Nothing consequential happens without explicit human approval.** Every pipeline stage that changes or groups a customer's data (cleaning, standardization, matching) produces a result that starts `pending_review` and requires an explicit `approve` call before it is treated as authoritative. Rollback is always a pure status transition, never a destructive operation — output files and change/decision rows are left untouched on rollback or rejection, because the source data was never touched in the first place. Automatic execution (the Module 12 scheduler) is scoped narrowly to *creating a pending run*, never to auto-approving its result.

**The execution engine is generic; behavior lives in handlers.** `app/worker/engine.py` claims, heartbeats, and completes a `TaskRun` without any knowledge of what a `sync`, `transform`, `standardize`, `match`, or `export` task actually does. Real behavior is added by writing a new `ExecutionHandler` and adding one line to `HANDLER_REGISTRY` (`app/worker/handlers/__init__.py`). Across five real handlers added over eight modules, the engine itself has never needed to change to support a new task type.

**Concurrency safety comes from fencing tokens, not trust.** Every claimed `TaskRun` gets a fresh, single-use `lease_token`. Every heartbeat and completion call must present the current lease token and expected status, or it has no effect. A worker whose lease was reclaimed by the reaper or another worker can never corrupt state it no longer owns, even if it is still running and reports a result late. The same shape (claim via `SELECT ... FOR UPDATE SKIP LOCKED` plus a guarded `UPDATE`) is reused, not reinvented, by the Module 12 scheduler for schedule-due-occurrence claiming.

**Every transformation is deterministic and pure.** The cleaning, standardization, and matching engines (`app/cleaning/`, `app/standardization/`, `app/matching/`) contain no I/O and no randomness — given the same input and configuration, they always produce the same output. This is what makes their output independently auditable and their determinism testable without a database or filesystem.

**Aggregates are always exact; detail persistence is bounded and deterministic.** Every audit table introduced since Module 6 (`CleaningChange`, `StandardizationChange`, `MatchDecision`, `MatchSkippedBlock`, `ExportRowExclusion`) caps the number of individual rows persisted per run, but the owning run's own summary counts (`total_changes_count`, `changes_by_rule`, `skipped_block_count`, `excluded_row_count`, and so on) are always the true, uncapped totals. When detail is capped, the retained sample is always the deterministic lowest-index rows, never a random or unspecified subset.

**Migrations are hand-written and verified against real PostgreSQL, never autogenerated and never SQLite-only.** SQLite is used exclusively for fast sandbox iteration; it is not authoritative for anything PostgreSQL-specific (`FOR UPDATE SKIP LOCKED`, native enum DDL, the 63-byte identifier limit). No module has merged to `main` without a real-PostgreSQL migration cycle and full regression pass.

**Extend, don't redesign.** New modules reuse existing patterns (the handler registry, the composite tenant FK, the approval state machine, bounded-detail audit tables) rather than introducing parallel mechanisms. Every module's "Key decisions" recorded in `PROJECT_CONTEXT.md` explicitly justifies any new pattern by showing why an existing one didn't fit, rather than defaulting to something new out of convenience.

**Evidence over assumption.** Architectural claims are verified empirically, not assumed. Index additions in Module 11 were added only after `EXPLAIN (ANALYZE, BUFFERS)` showed a measurable benefit at a deliberately seeded large-tenant scale; the SQLite `batch_alter_table` index-loss bug in Module 12 was caught by direct `sqlite_master` inspection before and after migration, not inferred; PostgreSQL identifier-length limits are checked by direct character counts before a migration is written, following a lesson Module 7 first had to learn the hard way in verification.

## 5. Business Problems Solved

**Inconsistent, error-prone manual data cleanup.** Organizations routinely receive operational data (customer lists, transaction exports, CRM dumps) as inconsistent CSV files — mixed types, blank-equivalent placeholder values, inconsistent casing and formatting, duplicate rows and duplicate underlying records under different spellings. Doing this cleanup manually is slow, inconsistent between staff members, and produces no audit trail of what was changed and why. The system's cleaning (Module 6) and standardization (Module 7) engines apply the same fixed, versioned rule set every time, and record every individual change made.

**No visibility into "why does this record look different from the source."** Deterministic rule engines alone are not enough if their output can't be explained. Every applied change is persisted with a rule-specific confidence value; every matching decision records which field and comparison produced it (`MatchDecision.blocking_key`, `reason`); every skipped comparison is recorded too (`MatchSkippedBlock`), so "why weren't these two records compared" is answerable, not just "why were these two records merged."

**Duplicate records accumulating across systems and imports.** The matching and deduplication engine (Module 8) groups likely-duplicate records using deterministic, explainable rules (exact and normalized-exact field comparison with configurable per-field weights) — never fuzzy, phonetic, or approximate matching, and never AI/ML-based similarity — so every match decision can be reproduced and explained to a human reviewer before anything is merged or excluded.

**No safe way to act on a reviewed result.** Finding duplicates is not useful without a safe way to produce a deduplicated deliverable. The export engine (Module 9) materializes an approved match result into an actual output file, with reserved provenance columns and a hard, permanent-failure collision policy that refuses to silently overwrite a customer's own data.

**Operational work with no queue and no download path.** Before Module 11, a human reviewer had no single place to see everything currently awaiting a decision across the whole pipeline; before Module 10, an approved result had no way to actually be retrieved. Both gaps are now closed structurally, not with ad hoc scripts.

**Manual re-triggering of recurring ingestion work.** Before Module 12, every `TaskRun` had to be created by an explicit API call — there was no way to say "sync this data source every hour" without a human or an external cron job calling the API repeatedly. The scheduler closes this gap for the one task type (`SYNC`) where "run this again on a fixed cadence" has an unambiguous meaning.

## 6. System Goals

- Enforce tenant isolation structurally everywhere data, credentials, or files are stored or accessed — never rely solely on an application-layer check that could be bypassed by a query bug.
- Execute data-operations work reliably under process crashes, network interruptions, and concurrent workers, with no duplicate side effects and no silently lost work (idempotency keys, lease fencing, the reaper, exponential-backoff retry).
- Require explicit human approval before any pipeline stage's output is treated as authoritative, and make every approval/rejection/rollback decision fully reversible in status (never destructive of underlying data).
- Make every automated decision — a cleaning change, a standardization change, a match/no-match decision, an export inclusion/exclusion — independently auditable after the fact, without needing to re-run the engine that made it.
- Keep the execution engine, approval state machine, and tenant-isolation mechanism stable and reused as new task types and pipeline stages are added, rather than growing a new parallel mechanism per module.
- Verify every migration and every concurrency-sensitive guarantee against real PostgreSQL before merge — SQLite sandbox results are directional, never sufficient on their own.
- Keep technology choices (web framework, ORM, specific database engine) documented as current implementation decisions distinguishable from the invariant architectural principles they implement, so a future substitution is a scoped, deliberate change rather than an assumed constant (see Section 23).

## 7. Non-Goals

These are explicit, current scope boundaries — most are documented in `PROJECT_CONTEXT.md`'s Known Limitations as deliberate, named limits, not oversights discovered after the fact.

- **No AI or machine-learning-based decision-making anywhere in the execution path.** Cleaning, standardization, and matching are 100% deterministic rule engines. AI-assisted correction is a named, deferred extension point (first identified as out of scope in Module 6's design), not built and not planned for the near term.
- **No fuzzy, phonetic, or approximate matching.** Module 8's `comparison_type` is limited to `exact` and `normalized_exact` by explicit design — this is a stronger constraint than "no AI," and is treated as equally important for explainability and reproducibility.
- **No field-level merge or enrichment across duplicate records.** Export (Module 9) performs row-level deduplication only — the canonical row (selected by a fixed, deterministic lowest-`row_index` rule) wins entirely; no field is ever combined from a non-canonical duplicate into the canonical record.
- **No cron or wall-clock/timezone-aligned scheduling.** Module 12's schedule format is a fixed elapsed-seconds interval only. "Every day at 9am in a given timezone" is not supported in this release.
- **No auto-chaining of pipeline stages.** The scheduler creates `SYNC` runs only; a human (or a future module) must explicitly trigger `TRANSFORM`/`STANDARDIZE`/`MATCH`/`EXPORT` via `source_task_run_id`. Automatic chaining was explicitly rejected for V1 because it would require either weakening the approval-gate architecture or guessing which prior run to build on.
- **No reviewer workflow beyond a read-only queue.** Module 11's `GET /review-queue` has no claim, assignment, ownership, locking, notification, or escalation-timer behavior. Every approve/reject/rollback action still goes through the existing per-run endpoints.
- **No non-CSV data source connectors are implemented for real execution.** `SourceType.POSTGRES`/`MYSQL`/`REST_API`/`S3`/`OTHER` all fail permanently under `SYNC` today; only `CSV_UPLOAD` is real.
- **No delivery mechanism for output files.** Every output-producing stage writes to tenant-scoped local storage only; there is no email, webhook, or object-storage delivery step.
- **No retention/cleanup policy for output files.** Rolled-back and rejected runs leave their output files on disk indefinitely.
- **No HTTP range/partial-download support.** Module 10 downloads always transfer the complete artifact in one request.
- **No customer-facing rate limiting anywhere in the API**, a project-wide gap, not specific to any one module.

## 8. Complete Production Architecture

The system is two independently deployable, independently scalable OS processes sharing one PostgreSQL database:

```
                          ┌───────────────────────┐
                          │      PostgreSQL 16      │
                          │  (single source of truth) │
                          └────────────┬────────────┘
                                       │
                 ┌─────────────────────┼─────────────────────┐
                 │                                            │
      ┌──────────▼──────────┐                     ┌───────────▼───────────┐
      │   FastAPI process    │                     │     Worker process      │
      │  (app.main:app)       │                     │  (python -m app.worker) │
      │                       │                     │                        │
      │  auth / data_sources  │                     │  run_forever() loop:    │
      │  tasks / review_queue │                     │   1. scheduler pass     │
      │  internal (metrics)   │                     │      (Module 12)        │
      │                       │                     │   2. claim_batch()      │
      │  Never executes a      │                     │   3. execute_one() per  │
      │  TaskRun itself —      │                     │      claimed run        │
      │  only creates/reads     │                     │   4. reaper pass        │
      │  rows.                 │                     │      (expired leases)    │
      └───────────────────────┘                     └────────────────────────┘
                 │                                            │
                 └─────────────── tenant-scoped ───────────────┘
                              filesystem roots:
                     CSV_INPUT_ROOT / CSV_OUTPUT_ROOT /
                CSV_STANDARDIZED_ROOT / CSV_EXPORTED_ROOT
                  (each: {root}/{organization_id}/...)
```

A worker crash or restart never affects API availability, and the two processes scale independently — more workers can be added under load without touching the API tier, and vice versa. This split is not incidental: the API process never executes a `TaskRun`'s handler logic itself; it only ever creates a `pending` row (or, for the scheduler, the worker process itself creates it) and reads results back. All actual execution happens in the worker process, behind the lease-fenced claim mechanism described in Section 12.

Within the worker process, `run_forever()` (`app/worker/runner.py`) is a single loop that, each iteration: runs the Module 12 scheduler pass if its poll interval has elapsed (creating due `TaskRun`s), claims a batch of pending runs via `claim_batch()`, executes each claimed run synchronously within that iteration via `execute_one()` (which starts a background heartbeat thread per run while its handler executes), and runs the reaper pass if its poll interval has elapsed. The scheduler pass runs *before* `claim_batch()` deliberately, so a run it creates this iteration can be claimed by the very same iteration's `claim_batch()` call rather than waiting a full poll interval. The reaper currently runs as a second pass in the same loop/process as an MVP simplification — its only dependency is a database session, so splitting it into its own process is a scoped, low-risk future change (see Section 28), not an architectural rewrite.

## 9. Request Lifecycle

A typical authenticated API request follows this path, using `POST /tasks/{id}/runs` as the representative example:

1. **Transport.** Uvicorn receives the HTTP request and hands it to the FastAPI application (`app.main:app`).
2. **Routing.** FastAPI dispatches to the matching path operation in `app/api/tasks.py` based on path and method.
3. **Authentication.** The `get_current_active_user` dependency (`app/api/deps.py`) runs first: it extracts the bearer token via `OAuth2PasswordBearer`, decodes and verifies the JWT (`app.core.security.decode_access_token`), loads the `User` row by the token's `sub` claim, and — critically — re-checks the token's `org_id` claim against that user's *current* `organization_id` in the database, not just trusting the token's embedded value. This defends against a token issued before a hypothetical future org transfer, and against any tampering that swapped `sub` without also swapping `org_id`. An inactive user is rejected with `403`; an invalid/expired/malformed token is rejected with a single generic `401` that does not distinguish the failure reason.
4. **Request validation.** The request body (if any) is parsed and validated against its Pydantic v2 schema (e.g. an optional `{"source_task_run_id": "<uuid>"}` body for a `TRANSFORM` task's run creation). Validation failures return `422` with field-level detail, before any handler code runs.
5. **Tenant-scoped lookup.** The endpoint loads the referenced `Task` filtered by both its id and the caller's `organization_id` in the same query — a cross-tenant reference is indistinguishable from a nonexistent one, and both return `404`.
6. **Domain validation.** Task-type-specific rules run next (e.g. a `TRANSFORM` task requires `source_task_run_id`; a non-`SYNC` task rejects `schedule_interval_seconds`). These return `400` for a well-formed but semantically invalid request, distinct from `422`'s "malformed request body."
7. **Server-derived field construction.** Fields the client must never control directly — `organization_id`, `task_id`, `triggered_by`, `idempotency_key` — are always set from the authenticated context or generated server-side, never accepted from the request body.
8. **Persistence.** The endpoint constructs the row (via the shared `create_task_run_record()` factory as of Module 12 — see Section 12) and commits within the request's own database session (`app.db.session.get_db`, one session per request, closed at the end of the request).
9. **Response serialization.** The persisted row is serialized through its `Read` Pydantic schema (`ConfigDict(from_attributes=True)`), which deliberately omits internal worker-ownership fields such as `lease_token`.
10. **No execution happens synchronously.** The API request returns as soon as the `pending` row is committed. Execution happens later, asynchronously, in the separate worker process — the API layer's job ends at "durably recorded the intent," never "performed the work."

Every endpoint in the system follows this same shape: authenticate → tenant-scope the lookup → validate → derive server-owned fields → persist → serialize. No endpoint has its own bespoke authentication or tenant-scoping mechanism (Section 22).

## 10. Data Lifecycle

Data moves through the system in a strict, approval-gated pipeline. Each stage's handler reads the immediately-prior stage's *approved* output (never any other stage's, and never a `pending_review` or `rejected` one) and writes to its own dedicated, tenant-scoped storage root, never overwriting or reopening a prior stage's file for writing.

```
 CSV_INPUT_ROOT/{org}/           source file, uploaded/placed by the tenant, read-only
        │
        ▼  SYNC  (CsvProfilingHandler, Module 5)
 DataProfile row                 immutable, one per TaskRun; no output file
        │  (requires DataProfile to exist)
        ▼  TRANSFORM  (CleaningHandler, Module 6)
 CSV_OUTPUT_ROOT/{org}/          cleaned file + CleaningRun + CleaningChange rows
        │  (requires CleaningRun.status == approved)
        ▼  STANDARDIZE  (StandardizationHandler, Module 7)
 CSV_STANDARDIZED_ROOT/{org}/    standardized file + StandardizationRun + StandardizationChange rows
        │  (requires StandardizationRun.status == approved)
        ▼  MATCH  (MatchHandler, Module 8)
 MatchRun / MatchGroup /         no output file — matching only groups existing
 MatchDecision / MatchSkippedBlock  records, never rewrites them
        │  (requires MatchRun.status == approved)
        ▼  EXPORT  (ExportHandler, Module 9)
 CSV_EXPORTED_ROOT/{org}/        final deduplicated deliverable + ExportRun +
                                  ExportRowExclusion rows
        │
        ▼  (Module 10)
 GET .../download                verified-then-streamed retrieval of any
                                  approved/rolled_back stage's output bytes
```

Each arrow in this diagram is a hard database-and-application-level gate, not a convention: a handler whose upstream requirement is not met (upstream run missing, or not `approved`) fails permanently (`PermanentExecutionError`), never proceeds with a best-effort guess. A stage's own run always starts `pending_review` and requires an explicit human `approve` call — via the fixed state machine `pending_review → approved/rejected → optionally rolled_back` — before the next stage's handler will accept it as valid upstream input. Rollback is always a pure status transition: the underlying output file and every change/decision row are left exactly as they were, because a rolled-back run is retained for audit/investigation, never treated as if it had never happened.

Entry into this pipeline happens one of two ways: a human calling `POST /tasks/{id}/runs` directly (with an optional `source_task_run_id` to chain a non-`SYNC` stage to a specific upstream run), or — for `SYNC` tasks only, as of Module 12 — the scheduler automatically creating a `pending` run when a task's `next_run_at` is due. Both paths converge on the same `create_task_run_record()` factory and the same execution engine; a scheduler-created run is distinguished only by `triggered_by IS NULL`, and is otherwise claimed, executed, and reported exactly like a manually created one.

## 11. FastAPI Architecture

**Current implementation:** FastAPI (0.115.6) with Uvicorn as the ASGI server, Pydantic v2 (2.10.5) for request/response validation and settings, and SQLAlchemy 2.0 (typed `Mapped`/`mapped_column` declarative style, 2.0.37) as the ORM/Core layer.

`app/main.py` is a thin application factory (`create_app()`): it configures structured logging, builds the `FastAPI` instance with environment-conditional `/docs`/`/redoc` exposure (disabled entirely when `APP_ENV=production`), and registers one router per domain area — `health`, `auth`, `data_sources`, `tasks`, `review_queue`, `internal`. Adding an API surface for a new module means adding one new router module and one `include_router()` call; no shared routing infrastructure needs modification.

**Layering.** Every endpoint module (`app/api/*.py`) follows the same internal structure: FastAPI path operation functions depend on `get_db` (a per-request SQLAlchemy `Session`, `app/db/session.py`) and one of `get_current_active_user`/`get_current_superuser` (`app/api/deps.py`) for authentication, validate the request via a Pydantic schema from `app/schemas/`, operate on SQLAlchemy models from `app/models/`, and — where an endpoint's logic is substantial and independently testable (e.g. Module 11's review-queue aggregation) — delegate to a pure, framework-independent function outside the endpoint module itself (`app/review_queue/query.py::fetch_review_queue()`) rather than embedding complex logic directly in the path operation function.

**Shared dependencies.** `PaginationParams` (`app/api/deps.py`) is the single pagination mechanism reused by every list endpoint (`limit`/`offset`, default 50, hard max 100, `422` — never silently clamped — if exceeded). `get_current_superuser` gates the one operational, cross-tenant-shaped endpoint in the system (`GET /internal/metrics`) — every other endpoint is scoped to the caller's own organization by construction.

**Response schemas never expose internal fields.** Every `*Read` Pydantic schema is built with `ConfigDict(from_attributes=True)` and is hand-curated to include only fields appropriate for an API consumer — `TaskRunRead` omits `lease_token`; `CleaningRunRead`/`StandardizationRunRead`/`ExportRunRead` omit `output_file_path` (removed in Module 10, the first and so far only field ever removed from a shipped response schema, done only after a repository-wide grep confirmed the narrow blast radius).

**Current implementation vs. core architecture.** The requirement that survives any future technology substitution is: one process serving a REST API over HTTP, backed by a relational database, with per-request tenant-scoped authentication and Pydantic-equivalent request/response validation. FastAPI, Uvicorn, and Pydantic are the current implementation of that requirement — chosen for async-native performance, first-class OpenAPI generation, and strong static typing integration — not an assumed permanent constraint. A future migration to a different Python web framework, or to gRPC/GraphQL for specific consumers, would need to preserve exactly the layering described above (authenticate → tenant-scope → validate → derive server-owned fields → persist → serialize) to remain compatible with this architecture.

## 12. Task Execution Engine

The execution engine (`app/worker/engine.py`, introduced in Module 4) is the single mechanism by which any `TaskRun`, regardless of task type, is claimed, executed, and completed. It has not changed in shape since Module 4 despite five real handlers and a scheduler being added on top of it since.

**Claiming.** `claim_batch(db, worker_id, batch_size)` selects up to `batch_size` pending, due (`next_retry_at` null or past) `TaskRun`s ordered by `created_at`, using `SELECT ... FOR UPDATE SKIP LOCKED` on PostgreSQL — concurrent workers polling simultaneously never block each other and never claim the same row, because a row another transaction has locked is simply skipped, not waited on. On SQLite (sandbox only — SQLite has no real row-level locking) this degrades to a plain `SELECT`, an explicitly flagged fidelity gap: SQLite results for claim-atomicity are indicative, never conclusive; only real-PostgreSQL verification actually proves it (and Module 12's concurrency tests do so with two genuinely concurrent threads and a `threading.Barrier`, not simulated). Each claimed row transitions `pending → running` via a guarded `UPDATE ... WHERE status = 'pending'`, generating a fresh `lease_token` (a UUID, not a stable worker identity) and a `lease_expires_at` derived from the task's own `timeout_seconds` or the worker's configured default.

**Lease fencing.** Every heartbeat (`heartbeat()`) and completion call (`complete_success()`/`complete_failure()`) must present the exact `lease_token` issued at claim time and require `status = 'running'` in its guarded `UPDATE`'s `WHERE` clause. If the lease was reclaimed (by the reaper on expiry, or in a hypothetical future by another worker), the row now carries a different `lease_token`, so the guarded update affects zero rows and the caller receives `LeaseLostError` — a clean, explicit signal to stop reporting results, never a corrupted or double-applied state change. This is enforced additionally at the database layer by `ck_task_runs_lease_consistency`: a `running` row must always carry both `lease_token` and `lease_expires_at`; every other status must carry neither.

**Idempotency.** Every `TaskRun` receives an `idempotency_key` (a UUID) once, at creation, that never changes across retries of that same row. Handlers must pass this value to any downstream system whose write they perform, so a retried execution (after a crash, a lost lease, or an explicit retry) cannot create a duplicate downstream effect. It is exposed read-only on `TaskRunRead`.

**Execution.** `execute_one()` (`app/worker/runner.py`) starts a background heartbeat thread (extending the lease on a fixed interval via its own independent database session), looks up the appropriate `ExecutionHandler` via `get_handler(task.task_type)` (`app/worker/handlers/__init__.py`'s `HANDLER_REGISTRY`), builds an `ExecutionContext` (the task run, task, data source, idempotency key, and a `CredentialProvider`), and calls `handler.execute(context)`. A handler signals a retryable failure by raising `TransientExecutionError` and a non-retryable one by raising `PermanentExecutionError` — the engine itself never tries to classify an error; that decision belongs entirely to the handler that produced it. Any other unexpected exception is treated as retryable and logged with a full traceback, never silently swallowed.

**Retries.** On a retryable failure with attempts remaining (`task_run.attempt_count < max_attempts`, per-`Task`-configurable via `max_attempts`, falling back to `WORKER_DEFAULT_MAX_ATTEMPTS`), the run is requeued to `pending` with `started_at`/`finished_at`/`error_message` reset to `NULL` and `next_retry_at` set using capped exponential backoff (`WORKER_RETRY_BASE_DELAY_SECONDS * 2^(attempt-1)`, capped at `WORKER_RETRY_MAX_DELAY_SECONDS`). `attempt_count` persists across the requeue — it is never reset — so the backoff and the attempts-remaining check both see the true cumulative attempt history.

**Timeouts and reaping.** A worker heartbeats a claimed run on `WORKER_HEARTBEAT_INTERVAL_SECONDS`, extending `lease_expires_at`. `app/worker/reaper.py::reap_expired_runs()` runs as a second pass in the same worker process loop, recovering any `running` row whose lease expired without a heartbeat (worker crash, lost connectivity) — reclaiming it exactly as a worker-reported failure would (requeue if attempts remain, else terminate as `failed`).

**Audit trail.** Every state transition (`claimed`, `requeued`, `succeeded`, `failed`, and the reaper's equivalents) is appended to `task_run_events` — append-only, never updated or deleted, retaining full untruncated error detail even after the mutable `TaskRun` row itself has moved past that attempt.

**Adding a new task type.** The entire extension surface is: write a new module implementing the `ExecutionHandler` protocol (`app/worker/handlers/base.py` — a single `execute(context) -> str | None` method, raising one of the two typed exceptions on failure), add one line to `HANDLER_REGISTRY`, and — if the new type requires an upstream approved run — check for and validate that in the handler itself (the pattern every real handler from Module 5 onward already follows). Nothing in `app/worker/engine.py` or `app/worker/runner.py` has ever needed to change to support a new task type across the five real handlers built so far.

## 13. Worker & Scheduler Architecture

The worker is a standalone OS process, started with `python -m app.worker`, running `run_forever()` (`app/worker/runner.py`) until terminated. It has its own `docker-compose.yml` service (`worker`), independent of the `app` (API) service, sharing only the database.

**The loop.** Each iteration of `run_forever()`: (1) if `scheduler_poll_interval_seconds` has elapsed since the last scheduler pass, run `run_due_schedules()`, defensively wrapped so an isolated scheduling failure can never crash the loop; (2) call `claim_batch()` and synchronously execute every claimed run via `execute_one()`; (3) if `reaper_poll_interval_seconds` has elapsed, run `reap_expired_runs()`; (4) if nothing was claimed this iteration, sleep `worker_poll_interval_seconds` before the next iteration. The scheduler pass deliberately runs *before* the claim step in the same iteration, so a run it creates can be claimed in the same pass rather than waiting a full poll cycle.

**The scheduler (Module 12).** `run_due_schedules(db, worker_id, batch_size)` (`app/worker/scheduler.py`) is a bounded loop of independent, single-task database transactions — not one whole-batch transaction — a deliberate architectural correction made during Module 12's design review specifically for fault isolation (one bad task's failure or rollback can never touch another already-committed task's progress within the same pass) and to keep each individual lock's duration minimal. For each pass, up to `batch_size` due tasks (active, `schedule_interval_seconds IS NOT NULL`, `next_run_at <= now`) are selected via `SELECT ... FOR UPDATE SKIP LOCKED`, one at a time; each selected task's row is advanced with a guarded `UPDATE ... WHERE next_run_at = <the value just read>` (a second concurrency-safety layer beyond `SKIP LOCKED` alone, catching the theoretical case where two passes raced on the same row), and — only if that guarded update actually affected a row — a new `TaskRun` is created via the shared `create_task_run_record()` factory with `triggered_by=NULL`. A pass-local `excluded_ids` set tracks tasks already handled this pass, preventing a single pathological task from starving other due tasks within the same bounded loop.

**Missed-schedule policy.** If a task's `next_run_at` has fallen behind (the worker was down, or the interval was very short), exactly one catch-up `TaskRun` is created — anchored to the *claim* time, not the original missed `next_run_at` — and `next_run_at` is advanced by exactly one interval from that claim time. There is never a "catch-up storm" of multiple backlogged runs created in a single pass.

**Schedule model.** `Task.schedule_interval_seconds` (nullable integer, SYNC-only) and `Task.next_run_at` (nullable timestamp) are the only fields that drive scheduling. The pre-existing `Task.schedule` free-text column (present since Module 3, originally intended for a future cron-like format) is explicitly deprecated: still readable and writable for backward compatibility, but never read by the scheduler and never reinterpreted — setting it alone produces no scheduling behavior whatsoever. Bounds on `schedule_interval_seconds` are enforced at two independent layers: a configurable, fail-fast-at-startup application-layer minimum/maximum (`MINIMUM_SCHEDULE_INTERVAL_SECONDS`/`MAXIMUM_SCHEDULE_INTERVAL_SECONDS`, never silently clamped — an out-of-bounds `Settings` value raises `ValidationError` at process startup), sitting above a fixed, non-configurable database `CHECK` floor of 30 seconds (`ck_tasks_schedule_interval_hard_floor`) that exists independently of application configuration as a last-resort backstop against a catch-up-storm risk from an extremely short interval.

**Metrics.** Four Module 12 Prometheus metrics (`task_scheduler_passes_total`, `task_scheduler_runs_created_total`, `task_scheduler_errors_total`, `task_scheduler_last_success_timestamp_seconds`) inherit a pre-existing, project-wide limitation rather than introducing a new one: they increment inside the worker process's own in-memory Prometheus registry, while `GET /internal/metrics` reads the FastAPI process's separate in-memory registry — the two processes never share counter state. This gap already existed for the four Module 4 execution-engine counters before Module 12; closing it (a metrics endpoint inside the worker process, or a push gateway) is unbuilt, project-wide infrastructure work, not scoped to any single module (Section 21, Section 28).

**Reaper placement.** The reaper currently runs as a second pass inside the same worker process/loop as an MVP simplification. It depends on nothing but a database session, so splitting it into its own process or service — for independent scaling or to decouple its poll cadence from the main claim loop — is a scoped, low-risk future change, not an architectural rewrite (Section 28).

## 14. AI Multi-Agent Architecture

This section requires an explicit framing decision, stated up front: **nothing in this system today is an AI agent in the machine-learning sense.** No component makes a probabilistic inference, calls a language model, or learns from data. Every decision made anywhere in the pipeline — a cleaning rule application, a standardization classification, a match/no-match decision — is deterministic, versioned, and fully reproducible from its inputs and configuration alone. This is a firm, repeatedly-reaffirmed project principle (Section 4, Section 7), not an implementation detail that happens to be true today.

What this project *does* have, and what "AI Data Operations Employee" actually refers to architecturally, is a **multi-agent-shaped pipeline of specialized, independently operating execution units**, each responsible for one well-defined stage of data-operations work: a sync agent (`CsvProfilingHandler`), a cleaning agent (`CleaningHandler`), a standardization agent (`StandardizationHandler`), a matching agent (`MatchHandler`), and an export agent (`ExportHandler`). Each is registered against a `TaskType` in `HANDLER_REGISTRY` and operates independently: it reads only its own well-defined upstream input (never reaching outside its stage's contract), performs its own bounded unit of work, and hands off a reviewable result rather than autonomously proceeding to the next stage. This is "multi-agent" in the software-architecture sense of the term (a registry of independent, single-responsibility execution units coordinated by a shared engine and a shared approval gate) — not in the sense of autonomous, LLM-driven agents making open-ended decisions.

**Why this framing matters for future modules.** The system's current architecture already provides the two things a genuine future move toward AI-assisted decision-making would need to slot into safely, without weakening any existing guarantee: (1) a handler registry that isolates one unit of specialized behavior per task type behind a stable `ExecutionHandler` interface, and (2) a mandatory human-approval gate that already sits between every stage's output and its being trusted by the next stage. If AI-assisted correction is ever built (it remains a named, deferred extension point since Module 6's design, not scheduled work), the correct integration point is a new handler (or a configurable mode within an existing one) whose *suggestions* still flow through the exact same `pending_review → approved/rejected` gate every deterministic stage already uses today — never a bypass of that gate, and never a change to the underlying deterministic engines that must remain available, explainable, and independently reproducible regardless of what any AI-assisted layer suggests on top of them.

**Current implementation vs. optional future alternative.** Current implementation: five deterministic handlers, statically registered, human-approved at every consequential stage. Optional future alternative (explicitly not built, not scheduled, and not implied by anything in this document): an additional handler type that produces *suggested* changes for human review, rather than applying changes directly — still bounded by the same registry-and-approval architecture, distinguished only by its suggestions requiring review rather than its actions requiring review, which is not actually a difference at the architecture level. Any such addition is a new module's own design/approval cycle, not a reinterpretation of this document.

## 15. Configuration & Tenant Isolation

**Configuration.** All runtime configuration is sourced from environment variables (or a local `.env` file in development) via `pydantic-settings` (`app/core/config.py::Settings`), never hardcoded per-environment. Settings are grouped by the module that introduced them (Application, Server, Database, Logging, Security, JWT/Auth, Worker/execution engine, CSV ingestion/profiling, cleaning, standardization, matching, export, scheduled task execution) and validated at process startup — an invalid combination (e.g. `MAXIMUM_SCHEDULE_INTERVAL_SECONDS` below `MINIMUM_SCHEDULE_INTERVAL_SECONDS`) raises `ValidationError` immediately via a `model_validator`, never silently clamping to the nearest valid value and never deferring the failure to first use. `get_settings()` is `lru_cache`d — settings are read from the environment exactly once per process.

**Tenant isolation — database layer.** Every domain table carries an `organization_id` column. Every tenant-scoped foreign key is a composite `(organization_id, X) → parent(organization_id, id)`, which requires the referenced parent to declare an explicit `UniqueConstraint(organization_id, id)` — a plain `id`-only foreign key is never used for a tenant-owned parent anywhere in this schema. This means a query that joins across two tenant-scoped tables cannot accidentally return a cross-tenant row: the foreign key itself would be violated if a child row's `organization_id` ever disagreed with its parent's. Soft-deleted (`is_active=False`) resources are excluded from default reads and behave identically to nonexistent resources for lookup purposes.

**Tenant isolation — API layer.** Every protected endpoint depends on `get_current_active_user`, which re-verifies the JWT's `org_id` claim against the loaded user's *current* `organization_id` in the database on every single request (Section 9), not just at token issuance. Every tenant-scoped lookup filters by `organization_id` in the same query that filters by resource id — never a two-step "load by id, then check org" pattern that could be bypassed by an incomplete guard. Cross-organization access always returns `404`, never `403`, so a caller cannot distinguish "exists in another org" from "does not exist anywhere."

**Tenant isolation — filesystem layer.** Every tenant-supplied file path (CSV input, cleaned output, standardized output, exported output) is resolved against that organization's own subdirectory of a server-controlled root (`CSV_INPUT_ROOT/{organization_id}/`, and the equivalent for each of the three output roots), never against the shared root directly. Traversal-escape checks (absolute-path rejection, `../` rejection) are layered *on top* of this per-tenant confinement, not used *instead* of it — this distinction was the actual B1 security fix in Module 5.1: the pre-fix code had traversal protection but resolved every organization's `file_path` against one shared root, meaning any org could reach any file if it knew or guessed the filename. Confinement to a tenant's own subdirectory closes that gap structurally.

**Tenant isolation — secrets layer.** `DataSource.connection_metadata` is scanned recursively against a secret-key denylist (`app/core/validation.py::find_secret_like_key`, matching keys like `password`, `token`, `api_key`, `secret`, and related patterns) and rejected with `422` if any secret-shaped key is present — this metadata table is never allowed to hold live credentials. Real credentials are set via a dedicated write-only endpoint (`PUT /data-sources/{id}/credentials`, no corresponding `GET` anywhere) and stored encrypted at rest (Fernet, `app/core/encryption.py`) in `data_source_credentials`, a table the execution engine and handlers never query directly — they depend only on the `CredentialProvider` interface (`app/worker/credentials.py`), so migrating to a managed secrets service later requires a new provider implementation and a one-line swap in `app/worker/runner.py`, no change to engine or handler code.

## 16. Database Architecture

**Current implementation:** PostgreSQL 16 (`postgres:16-alpine` in production/Docker), SQLAlchemy 2.0 typed declarative ORM/Core, Alembic hand-written (never autogenerated) linear migration history, currently at 12 revisions ending at `d5e6f7a8b9c0`. SQLite is used exclusively as a fast local sandbox/CI target and is never authoritative for PostgreSQL-specific behavior.

**Schema shape.** Every domain table: a UUID primary key, `organization_id` (composite-FK-scoped as described in Section 15), `created_at`/`updated_at` timestamps, and — for any resource that can be deactivated — `is_active` plus a partial unique index on `lower(trim(name))` scoped to active rows only (soft delete frees the name for reuse without a hard delete). Native PostgreSQL enum types (`source_type_enum`, `task_type_enum`, `task_run_status_enum`) are owned exclusively by Alembic migrations — `CREATE TYPE`/`DROP TYPE` never happens anywhere outside `database/alembic/versions/`, and every ORM model's corresponding `Enum(...)` object sets `create_type=False` so `Base.metadata.create_all()` (used by the test suite's table-creation fixture) never independently attempts to create these types. This split ownership was learned the hard way in Module 4: leaving `create_type` at SQLAlchemy's default caused a real `DuplicateObject` migration bug when both `create_all()` and Alembic tried to create the same type.

**Extending vs. creating enums.** Two different migration shapes exist in this project's history: creating a brand-new native enum type (every enum-touching migration before Module 7), and extending an existing one via `ALTER TYPE ... ADD VALUE` (Module 7's `standardize` and Module 8's `match` additions to `task_type_enum`). The latter must run inside Alembic's `autocommit_block()` — PostgreSQL disallows using a newly added enum value within the same transaction that added it — and has no `DROP VALUE`; a downgrade leaves the added label in place permanently, which is expected, standard PostgreSQL behavior, not a bug requiring a workaround.

**Identifier length.** PostgreSQL enforces a hard 63-byte identifier limit (`NAMEDATALEN`) that SQLite does not enforce at all — this class of bug is invisible to SQLite-only verification and was only caught by real-PostgreSQL verification in Module 7 (an auto-derived FK constraint name on `standardization_column_mappings` exceeded the limit). Every migration since Module 8 pre-measures hand-written constraint/index names against this limit before the migration is written, not after a verification failure surfaces it.

**SQLite-specific migration constraints.** SQLite cannot `ALTER TABLE ADD CONSTRAINT` — adding a `CHECK` constraint to an existing table requires Alembic's `batch_alter_table`, which performs a full table-copy-and-recreate under the hood. This recreate step cannot reflect expression-based partial indexes, silently dropping them — a real bug caught and fixed during Module 12's migration authoring (`ix_tasks_org_name_active` was being silently dropped), now guarded by an explicit drop-and-recreate step around every `batch_alter_table` block and a dedicated PostgreSQL regression test.

**Nullable-scope uniqueness.** Any configuration table whose uniqueness key includes a nullable "applies to everything" column (e.g. `data_source_id` on `standardization_column_mappings`, `match_rule_sets`) requires two partial unique indexes, not one, because standard SQL `NULL != NULL` semantics would otherwise let duplicate NULL-scoped rows through undetected by a single index. This pattern, established in Module 7, was reused without modification in Module 8.

**Bounded audit persistence.** Every per-cell/per-decision audit table (`CleaningChange`, `StandardizationChange`, `MatchDecision`, `MatchSkippedBlock`, `ExportRowExclusion`) caps the number of individual rows persisted per run via a dedicated configuration setting, while the owning run's own aggregate summary fields remain exact regardless of the cap (Section 4). This is the standing pattern for any future module needing to persist potentially-large per-item detail without unbounded table growth.

## 17. Validation Pipeline

Validation happens at multiple independent layers, each catching a different class of problem, and no layer is treated as sufficient on its own:

**Pydantic schema validation (request shape and type correctness).** Every request body is validated against a Pydantic v2 schema before any endpoint code runs — malformed JSON, wrong types, out-of-range values (e.g. `schedule_interval_seconds` below the configured minimum, validated via a dynamic-bounds `field_validator` that reads current `Settings` rather than a hardcoded literal) all fail with `422` before touching the database.

**Domain/business-rule validation (semantic correctness within the endpoint).** Rules that depend on more than the shape of a single field — a `TRANSFORM` task requiring `source_task_run_id`, a non-`SYNC` task rejecting `schedule_interval_seconds`, a task-type change that would leave a scheduled non-`SYNC` task in an invalid state — are checked explicitly in the endpoint layer and return `400`, distinct from `422`'s "the request itself is malformed."

**Secret-key denylist validation (content-shape safety).** `connection_metadata` on a `DataSource` is recursively scanned against `SECRET_KEY_DENYLIST` before persistence; any key that looks credential-shaped is rejected with `422`. This is explicitly documented as a best-effort static check, not a security guarantee — real secret storage is `DataSourceCredential`, not this field.

**Database constraint validation (the final, non-bypassable layer).** Every invariant that must hold regardless of which code path wrote a row is additionally enforced by a database `CHECK`, `UNIQUE`, or foreign-key constraint — `ck_task_runs_lease_consistency`, `ck_tasks_schedule_interval_hard_floor`, `ck_tasks_schedule_consistency`, the composite tenant-FK pattern itself. This layer exists specifically because application-layer validation can have bugs or be bypassed by a direct database write (as Module 12's own concurrency test explicitly proves by bypassing the Pydantic-layer minimum and confirming the database-layer floor independently rejects it) — the database is the last line of defense, never assumed to be redundant with the application layer above it.

**Deterministic pipeline-stage validation (content-shape correctness within the cleaning/standardization/matching engines).** Within the pure rule engines themselves, "validation" takes the form of type-coercion-toward-inferred-type and structural-issue detection (blank/duplicate headers, ragged rows, inconsistent column types) — never a hard rejection of the input file, since the whole point of these stages is to improve messy input, not merely reject it (Section 10).

No validation layer is a substitute for another: schema validation catching a malformed request does not mean domain validation can be skipped, and domain validation catching a business-rule violation does not mean the database constraint is redundant — each layer has caught a real bug the others did not, across this project's actual history (the SQLite `batch_alter_table` index bug, the Module 4 enum `create_type` bug, the Module 10 file-descriptor leak).

## 18. Audit & Rollback

**The approval state machine.** Every pipeline stage that changes or groups customer data (`CleaningRun`, `StandardizationRun`, `MatchRun`, `ExportRun`) follows the identical fixed state machine, first established in Module 6 and reused without modification in every subsequent stage:

```
pending_review ──approve──▶ approved ──rollback──▶ rolled_back
      │
      └──reject──▶ rejected
```

Any attempt to transition from a state other than the one a given action requires (e.g. approving an already-`rejected` run) returns `409 Conflict`. This state machine is enforced entirely through the existing per-run endpoints in `app/api/tasks.py` — Module 11's review queue reads this same state, never duplicates or caches it, and an item leaves the queue the instant its authoritative status changes underneath it, proven end-to-end by a dedicated test.

**Rollback is never destructive.** Rolling back an `approved` run to `rolled_back` is a pure status transition. The output file on disk and every associated detail row (`CleaningChange`, `StandardizationChange`, `MatchDecision`, `ExportRowExclusion`) are left completely untouched — this is possible specifically because the source file was never opened for writing by any handler in the first place, and the current stage's output already lives at a separate, dedicated path from the moment it was written. A `rolled_back` artifact remains downloadable (Module 10) for audit/investigation purposes, with `X-Artifact-Run-Status` always disclosing its true status so a client can never mistake it for current authoritative output.

**Append-only execution audit.** `task_run_events` records every claim/heartbeat-driven-requeue/success/failure/reap transition, and is never updated or deleted — even after a `TaskRun` row itself has moved past a given attempt (e.g. been requeued and its `error_message` cleared), the corresponding event row retains the full untruncated detail from that attempt.

**Download audit.** `artifact_download_events` (Module 10) records the complete lifecycle of every *authorized* download attempt — `started` transitioning exactly once to `completed`/`integrity_failed`/`file_missing`/`stream_failed` — with a small, controlled `failure_reason_code`, never raw exception text or a filesystem path. Unauthorized, cross-organization, nonexistent, `pending_review`, or `rejected` download attempts create no audit row at all, by design (they are rejected before any resource is identified as existing).

**Matching-specific audit depth.** Because matching decisions are the hardest to explain after the fact, Module 8 added two audit dimensions beyond what Modules 6/7 needed: `MatchDecision.blocking_key` records which field's value grouped two records into the same comparison block (null for Stage 1 exact-row matches, which use no blocking), and `MatchSkippedBlock` records a bounded, deterministic sample of row indices for any block that exceeded the configured size cap and was therefore never compared at all — closing the "why weren't these two records compared" question that a naive audit trail (which only records decisions actually made) cannot answer.

**Integrity verification before delivery.** Every artifact download (Module 10) re-verifies the complete file's SHA-256 via bounded chunked reads through a single `open()` call *before* any response byte is sent to the client — a hash mismatch sends zero bytes and fails with `500`, never a partially-corrupted stream. This was a deliberate architectural correction during Module 10's design review: an initial "hash while streaming" proposal was rejected because corrupted bytes could reach the client before a mismatch was ever detected.

## 19. Error Handling

**Two typed exceptions carry the entire retry-vs-terminal decision.** Every `ExecutionHandler` communicates failure exclusively through `TransientExecutionError` (expected to succeed on retry — network blips, timeouts, temporary unavailability) or `PermanentExecutionError` (will never succeed on retry — bad configuration, invalid input, an unsupported operation, a missing/unapproved upstream run). The engine itself never inspects an exception's message or type beyond this binary classification — the decision belongs entirely to the handler that raised it, since only the handler has the domain knowledge to make it correctly.

**Unexpected exceptions default to retryable, never silently swallowed.** Any exception a handler raises that is neither of the two typed errors is logged with a full traceback (`logger.exception`) and treated as retryable — the conservative default, since an unclassified failure might genuinely be transient, and a spuriously-permanent classification would discard work that could have succeeded on retry. It is never caught and discarded without a trace.

**API-layer error codes are consistent and deliberate, not ad hoc.** `401` — invalid, missing, or expired authentication, always a single generic message that does not distinguish wrong-org/wrong-email/wrong-password (login) or the specific reason a token failed (auth middleware), to avoid leaking which factor was wrong. `403` — an authenticated but inactive user, or an authenticated user lacking a required privilege (superuser-only endpoints) — never used for cross-tenant access, which is always `404` instead, so a caller cannot distinguish "exists but you can't see it" from "doesn't exist." `404` — resource not found, including every cross-organization reference, indistinguishable by design. `409` — a state-machine conflict (approving an already-decided run, a colliding organization slug at registration). `422` — malformed request shape or a value failing schema-level validation (including denylisted secret-shaped keys, out-of-bounds schedule intervals). `400` — a well-formed request that is nonetheless semantically invalid for domain reasons (missing `source_task_run_id` for a `TRANSFORM` task, a schedule interval on a non-`SYNC` task). `500` — reserved for genuine server-side failures the caller could not have prevented (an artifact integrity mismatch caught before any bytes are sent, an unexpected internal exception) — never used as a catch-all for a validation failure that should have been caught earlier.

**Lease-loss is a first-class control-flow signal, not an error to work around.** `LeaseLostError` propagating out of a heartbeat or completion call is the engine's way of saying "this row is no longer yours" — every caller (the heartbeat thread, `execute_one()`'s completion-reporting code) treats it as an unconditional signal to stop immediately, never to retry the same completion call or attempt to force the update through.

**Permanent failures at pipeline boundaries fail loudly and immediately, never degrade to a best-effort partial result.** A handler whose upstream requirement is unmet (missing or non-approved prior stage), or whose input violates a hard invariant (a reserved provenance-column-name collision in Module 9, an unsupported `source_type` under `SYNC`), raises `PermanentExecutionError` before any file write or persistence — never attempting a partial write and hoping the human reviewer catches the problem later.

## 20. Retry & Recovery

**Retry is a property of the execution engine, available uniformly to every handler, not something each handler implements itself.** A handler's only involvement in retry behavior is choosing which of the two typed exceptions to raise (Section 19); everything else — attempt counting, backoff calculation, requeue mechanics — is handled identically by `app/worker/engine.py::complete_failure()` regardless of task type.

**Exponential backoff with a hard cap.** Retry delay is `WORKER_RETRY_BASE_DELAY_SECONDS * 2^(attempt_count - 1)`, capped at `WORKER_RETRY_MAX_DELAY_SECONDS` (defaults: 30 seconds base, 900 seconds/15 minutes cap) — a retry storm immediately after a transient outage is naturally spread out, and a persistently failing task never backs off indefinitely into practical irrelevance.

**Per-task-configurable limits, with a system-wide default.** `max_attempts` and `timeout_seconds` are configurable per-`Task` (`null` falls back to the worker's global `WORKER_DEFAULT_MAX_ATTEMPTS`/`WORKER_DEFAULT_TIMEOUT_SECONDS`), so a task known to depend on a flaky upstream system can be given more attempts or a longer timeout without changing global worker behavior for every other task.

**Recovery from a crashed or unresponsive worker is lease-driven, not heartbeat-driven-alone.** If a worker process crashes mid-execution, its heartbeat thread simply stops — no explicit "I crashed" signal is needed. The reaper independently discovers the abandoned `running` row once its `lease_expires_at` passes with no further heartbeat, and reclaims it exactly as if the worker had reported a failure itself: requeued if attempts remain, else terminated as `failed`. This means recovery time is bounded by the lease timeout, not by any crash-detection mechanism that could itself fail.

**No in-process cancellation of work already in progress.** This is an explicit, named limitation, not an oversight: `WORKER_DEFAULT_TIMEOUT_SECONDS` governs lease-based *reclaim* by the reaper, not *interruption* of a handler already executing. A pathologically slow handler run is not stopped mid-flight — it continues running in its original worker process (now orphaned from the row it was working on, since the reaper has reassigned that row) until it finishes or the process itself is killed. A future module addressing this would need genuine handler-level cancellation support (e.g. a cooperative cancellation token threaded through `ExecutionContext`), which does not exist today.

**Idempotency is what makes retry safe, not merely functional.** Because every `TaskRun` carries a stable `idempotency_key` across all of its retries, and because every real handler either checks for an existing result by a unique constraint before writing (the "unique-constraint-plus-refetch" pattern established in Module 5 and reused by every handler since) or performs no external side effect that could duplicate, a retried run is guaranteed to produce the same end state as if it had succeeded on its first attempt — never a duplicate profile, cleaning run, standardization run, match run, or export.

## 21. Logging & Observability

**Structured logging.** All logging goes through Python's standard `logging` module, configured centrally by `app/core/logging.py::configure_logging()` (called once, at process startup, by both the API and worker entrypoints). Output format is environment-conditional: `json` (via `python-json-logger`) in production for machine parsing by a log aggregator, `console` in development for human readability. `LOG_LEVEL` is configurable per environment.

**Docstrings and log messages explain why, not just what.** This is a deliberately maintained convention across every module: a log line like a token `org_id` mismatch warning includes enough context (user id, token org, actual org) to investigate without needing to reproduce the request.

**Metrics.** Prometheus-format metrics are exposed via `GET /internal/metrics` (superuser-only, since it exposes cross-tenant, org-agnostic system state inappropriate for a general authenticated user). Eight counters/gauges/histograms exist today: four from Module 4's execution engine (`task_engine_tasks_claimed_total`, `task_engine_tasks_completed_total`, `task_engine_tasks_failed_total`, `task_engine_tasks_retried_total`, plus the `task_engine_execution_duration_seconds` histogram and `task_engine_queue_depth` gauge), and four from Module 12's scheduler (`task_scheduler_passes_total`, `task_scheduler_runs_created_total`, `task_scheduler_errors_total`, `task_scheduler_last_success_timestamp_seconds`).

**A known, disclosed, project-wide observability gap.** `/internal/metrics` runs inside the FastAPI process and reads that process's own in-memory Prometheus registry. Every counter that increments inside the worker process (all eight listed above except `task_engine_queue_depth`, which is recomputed fresh from a direct database query on every request rather than read from an in-memory counter) is therefore invisible through this endpoint — the two processes hold independent copies of the same registry object with no shared state. This gap existed from Module 4 onward and was explicitly re-disclosed, not silently repeated, when Module 12 added four more metrics subject to the identical limitation. Closing it — a metrics endpoint inside the worker process itself, or a push-gateway pattern where the worker actively pushes its counters to a shared collector — is unbuilt, project-wide infrastructure work; it is not scoped to any single module and should not be treated as "fixed" by any future module unless that module's design explicitly addresses this exact gap.

**Audit tables as a secondary observability surface.** Beyond metrics and logs, this system's append-only audit tables (`task_run_events`, `artifact_download_events`, and every stage's change/decision detail tables) serve as a durable, queryable observability surface that survives process restarts and log rotation — `GET /review-queue` (Module 11) is itself best understood as an operational observability endpoint built entirely on top of these existing audit tables, not a new data source.

## 22. Security Architecture

**Authentication.** Multi-tenant JWT authentication (Module 2). A user belongs to exactly one organization; because email is unique only *within* an organization, login requires three fields (organization slug, email, password), not two. Passwords are hashed with bcrypt (never reversible, never logged); the 72-byte bcrypt input limit is enforced at the schema layer with a hard rejection (`422`), never silent truncation — `hash_password()` itself treats exceeding the limit as an unreachable programming error if reached, since schema validation should always catch it first. Login failures for wrong organization slug, unknown email, or wrong password all return the identical generic `401`, deliberately not distinguishing which factor was wrong, to avoid confirming account existence to an unauthenticated caller. JWTs encode both the user id (`sub`) and `org_id`; both are re-verified against the database on every single request (Section 9, Section 15), never trusted from the token alone for the request's duration.

**Authorization.** Two dependency-based gates cover the entire system: `get_current_active_user` (any authenticated, active user, scoped to their own organization by every subsequent query) and `get_current_superuser` (additionally requires `is_superuser`, gating only the one cross-tenant-shaped endpoint, `/internal/metrics`). There is no endpoint-specific or ad hoc authorization mechanism anywhere in the codebase to keep track of — every protected endpoint uses one of these two dependencies, never a bespoke check.

**Secrets management.** Two distinct classes of sensitive data are handled differently, deliberately: `connection_metadata` (non-secret configuration, denylist-scanned and rejected if secret-shaped, Section 17) versus `DataSourceCredential` (real credentials, Fernet-encrypted at rest, accessed only through the `CredentialProvider` abstraction, write-only at the API layer with no corresponding `GET`). `CREDENTIAL_ENCRYPTION_KEY` is required and unconditionally enforced in production (`APP_ENV=production`); its absence in production is a hard startup-time failure for any credential operation, never a silent fallback. A local-development-only insecure fallback key exists explicitly and only for `APP_ENV != production`, with a loud warning logged every time it is used. The application's own JWT-signing `SECRET_KEY` follows the same pattern: an insecure default exists for local development convenience only and must be replaced with a real 32+ byte random value before any production deployment — this is documented, not silently assumed.

**Tenant isolation as a security control, not just a data-modeling convenience.** As detailed in Section 15, isolation is enforced at the database (composite FKs), API (re-verified `org_id` on every request), and filesystem (per-tenant subdirectories, never a shared root) layers simultaneously — a failure at any single layer does not by itself produce a cross-tenant data leak, because the other layers remain in place independently.

**Integrity verification before delivery.** Every downloaded artifact (Module 10) is fully re-hashed and compared against its stored SHA-256 before any response byte is sent, closing the window where corrupted or tampered bytes could otherwise reach a client undetected (Section 18).

**Concurrency safety as a security-relevant property.** The lease-fencing mechanism (Section 12) prevents a specific class of correctness bug — a stale, reclaimed worker corrupting state it no longer owns — that would otherwise be a genuine data-integrity risk under concurrent execution, not merely a performance concern.

**Explicit, disclosed gaps.** No rate limiting exists anywhere in the API (a project-wide gap, not module-specific). Sandbox verification currently runs on Python 3.10 while the production Docker image targets Python 3.13 — these have not been cross-verified against each other in this project's history and should be confirmed before a first real production deployment. Both gaps are recorded here deliberately, not omitted, consistent with this project's standing practice of disclosing known limitations rather than implying completeness.

## 24. Coding Standards

The following standards are actively enforced across every completed module and are binding on Module 13 and beyond. Each is stated with the concrete precedent that established it.

- **SQLAlchemy 2.0 typed declarative style only** (`Mapped[...]`/`mapped_column`) — no legacy `Column`-only style anywhere in the codebase.
- **Native PostgreSQL enums must use `sqlalchemy.dialects.postgresql.ENUM(..., create_type=False)`, never generic `sqlalchemy.Enum`** — the generic class silently discards `create_type`, which caused a real production-shaped `DuplicateObject` migration bug in Module 4. Alembic migrations are the sole owner of `CREATE TYPE`/`DROP TYPE`.
- **Small, internal, worker/config-owned value sets are plain `String` columns, not native Postgres enums** (`TaskRunEvent.event_type`, `CleaningRun.status`, `StandardizationRun.status`, `MatchRun.status`, `ExportRun.status`) — native enums are reserved for values with real external/API meaning (`SourceType`, `TaskType`, `TaskRunStatus`).
- **Every tenant-scoped table declares `organization_id`, the composite FK pattern, and — if it is a composite-FK target — an explicit `UniqueConstraint(organization_id, id)` at creation time**, not added reactively after a child table's FK fails. This was missed once (Module 6's `CleaningRun`) and is now checked proactively for every new parent table.
- **Soft delete via `is_active`, never a hard `DELETE`, for anything another row might reference.**
- **Docstrings explain why, not just what** — a deliberate, consistently maintained convention; new code should match it.
- **Pydantic v2 schemas use `ConfigDict(from_attributes=True)`; response models never expose internal worker-ownership fields** (`lease_token` is deliberately absent from `TaskRunRead`).
- **PostgreSQL identifier names have a hard 63-byte limit, silently unenforced by SQLite** — hand-written constraint/index names derived from long table names must be measured before the migration ships (established reactively by Module 7's verification failure, applied proactively by every module since).
- **Extending an existing native enum is a different migration shape from creating a new one** — requires `autocommit_block()`, has no `DROP VALUE`, and a downgrade leaves the added label in place permanently by design.
- **Deterministic blocking, not full pairwise comparison, is the standing pattern for any O(n²)-shaped comparison workload** — bucket candidates by a bounded key before comparing, and skip (never silently truncate) any bucket exceeding a configured size cap, recording the skip explicitly.
- **Bounded, deterministic detail sampling is the standing pattern whenever an aggregate must stay cheap but some inspectable detail should survive a cap** — exact aggregate count, capped-and-deterministic (never random) detail rows.
- **Partial unique indexes (two, not one) for any nullable-scope uniqueness key** — reach for this exact shape, established in Module 7 and reused in Module 8, rather than inventing a new one.
- **Reserved system columns injected into a customer-facing artifact must use a namespaced prefix and fail permanently on collision**, checked before any write or persistence, never silently renamed/suffixed/overwritten (established by Module 9's `__aiops_` prefix).
- **Run-specific timestamps must stay out of deterministic output artifacts** — persist wall-clock data on the database row, never inside a file whose determinism must be guaranteed.
- **File self-description metadata does not substitute for reading the file when true content-integrity verification is required** — state this distinction explicitly in any module adding similar metadata.

## 25. Testing Standards

- **SQLite is the sandbox/CI-speed verification target; it is never authoritative for anything PostgreSQL-specific** (`FOR UPDATE SKIP LOCKED`, native enum DDL, the 63-byte identifier limit, native `CHECK` constraint enforcement on a migration-local enum column). Real PostgreSQL verification is the final word before any merge, run explicitly as its own phase in every module's validation process, never skipped or assumed equivalent to a passing SQLite run.
- **Every new API endpoint requires explicit tests for:** the success path, cross-organization `404`, not-found `404`, and (where applicable) invalid-state-transition responses. No endpoint ships without this full set — established as the standard after the profile endpoint initially shipped in Module 5 without it, caught and fixed in 5.1.
- **Every new worker handler requires an idempotency test** proving a retried run does not duplicate its side effect, using the unique-constraint-plus-refetch pattern.
- **Any handler touching tenant-supplied file paths requires both a cross-tenant-isolation test and a traversal-escape test** — established as two genuinely different failure modes, both required, by the Module 5.1 (B1) fix.
- **A self-referential FK on any table requires the shared test-cleanup fixture to null out self-referencing columns before its bulk-delete pass** — a plain per-table `DELETE` can otherwise violate a `RESTRICT` FK depending on row order within one statement.
- **Genuine concurrency guarantees (claim-atomicity, scheduler duplicate-prevention) must be proven against real, concurrent PostgreSQL sessions** — real threads, independent database sessions, and an explicit synchronization barrier (`threading.Barrier`) where the guarantee under test is about simultaneous access, not inferred from sequential SQLite test execution, which cannot exercise real row-level locking at all.
- **A pure, framework-independent query/aggregation function is unit-tested directly, without a running API or `TestClient`** — reserve API-level tests for what is genuinely HTTP-boundary-specific (auth, end-to-end worker-produced data, query-param validation, response shape). Established by Module 11's `fetch_review_queue()` split between 22 direct unit tests and a smaller API-level test file.
- **Deterministic pagination requires a secondary, always-unique sort key whenever the primary sort column can plausibly tie** — PostgreSQL's `now()` is the transaction timestamp, constant across every statement in one transaction, so rows written together can share an identical `created_at`.
- **An independent, adversarial-minded review pass after implementation (not just before it) is standard practice for any module with nontrivial query/filter logic**, not a one-off — introduced in Module 11, where it found a real correctness gap (the pagination-determinism issue above) that the first review's own checklist had already named but the initial tests had not actually covered.
- **Real-PostgreSQL verification should explicitly attempt invalid tenant references and `CHECK` violations, not just run the migration cycle and test suite** — Module 9 established directly attempting distinct constraint violations (invalid status, invalid format version, negative size, the row-count invariant, cross-organization composite-FK references) against real Postgres and confirming rejection at the database layer; this is now the standard extra step, not a one-off.
- **Current baseline: 580 tests passing** (526 baseline through Module 11 plus 54 added in Module 12), on both SQLite and real PostgreSQL 16, full suite re-run required before every merge to `main`.

## 26. Deployment Architecture

**Current implementation.** `docker-compose.yml` defines three services: `db` (`postgres:16-alpine`, with a health check gating dependent services), `app` (the FastAPI process, built from `docker/Dockerfile`, a multi-stage build running as a non-root user, exposing port 8000), and `worker` (built from the identical `docker/Dockerfile`, running `python -m app.worker` as its container command instead of the API server). Both `app` and `worker` depend on `db`'s health check passing before starting, and both read `DATABASE_URL` pointed at the `db` service by its Compose network name. This is a single-host, single-database-instance deployment shape — appropriate for the project's current stage, not yet a multi-node production topology.

**Migration deployment.** Alembic migrations are applied via `alembic upgrade head`, run explicitly (currently a manual documented step, not yet wired into an automated container-startup hook) before the application or worker processes are expected to operate against a given database. `scripts/wait_for_postgres.py` provides a startup dependency check independent of Docker Compose's own health check, useful for non-Compose deployment contexts.

**Independent scalability by design, not yet exercised in this deployment shape.** Because the API and worker are already separate processes/containers sharing only the database, horizontally scaling the worker tier (more containers claiming from the same `pending` queue, safely, via the lease-fencing mechanism) or the API tier independently is architecturally supported today — `docker-compose.yml`'s current single-instance-per-service shape is a deployment-configuration choice, not an architectural ceiling. Scaling either tier is a matter of running more container instances of the existing image with the existing `DATABASE_URL`, not a code change.

**Current implementation vs. optional future alternatives.** Current implementation: Docker Compose, single-host, three services, manual migration step. Optional future alternatives, none built and none currently scheduled: Kubernetes (Deployments for `app` and `worker` with independent replica counts and HPA-driven autoscaling, a Job or init-container for migrations, a managed PostgreSQL service rather than a stateful in-cluster one); a managed container platform (ECS, Cloud Run) for the API tier paired with a separately-scaled worker fleet; CI/CD automation for the currently-manual migration-and-release sequence described in this project's own Git Workflow (Section 27 references the same discipline). None of these require a change to the application code itself — the two-process, shared-database architecture is already the right shape for any of them; only the orchestration layer around it would change.

## 28. Future Enterprise Architecture

This section describes directions that are consistent with the architecture already in place and would not require redesigning any completed module, but are explicitly **not built, not scheduled, and not implied to be imminent** by their presence here. They are recorded so that if and when they become relevant, they can be evaluated against this document rather than invented from scratch under time pressure.

**Horizontal worker scaling.** The lease-fencing claim mechanism (Section 12) already supports an arbitrary number of concurrent worker processes safely — this is a deployment-configuration change (more container instances), not an architectural one, and is the most immediately available scaling lever if execution throughput ever becomes a bottleneck.

**Splitting the reaper into its own process/service.** Currently an MVP simplification (a second pass in the same worker loop); its only dependency is a database session, so decoupling its poll cadence and failure domain from the main claim/execute loop is a scoped change.

**Closing the metrics-visibility gap.** Either a metrics endpoint inside the worker process itself, or a push-gateway pattern where the worker actively reports its counters to a shared collector rather than relying on `/internal/metrics`' passive, same-process read. This is project-wide infrastructure work affecting every counter introduced since Module 4, not specific to any single future module (Section 21).

**A managed secrets service.** The `CredentialProvider` abstraction (Section 15) exists specifically so that migrating from `DatabaseCredentialProvider` (Fernet-encrypted, application-layer) to a managed service (Vault, AWS/GCP Secrets Manager) requires writing one new provider implementation and a one-line swap in `app/worker/runner.py` — no change to the execution engine or any handler.

**Additional data source connectors.** The `SourceType` enum already reserves values (`POSTGRES`, `MYSQL`, `REST_API`, `S3`) beyond the one currently implemented (`CSV_UPLOAD`). Adding a real connector for one of these is architecturally the same shape as adding a new task-type handler (Section 12) — new, isolated code behind an existing interface, not a change to the engine.

**Streaming/incremental ingestion.** Today's CSV loading reads the entire file into memory up to a configured size bound. A genuinely large-file-capable ingestion path (chunked/streaming reads, incremental profiling) is a scoped extension to `app/profiling/csv_loader.py` and `app/profiling/csv_profiler.py`'s current pure-function contract, not a redesign of the pipeline around it.

**A retention/cleanup policy for output files.** No stage automatically deletes an output file on rollback, rejection, or age — a future module implementing retention would need to reconcile deletion against the "rollback is never destructive" principle (Section 18): deleting a file is a materially different guarantee than the current status-only rollback, and any such module must state explicitly which guarantee it is changing, not silently narrow the existing one.

**AI-assisted correction as a bounded extension, not a redesign.** As detailed in Section 14, the correct integration shape for any future AI-assisted capability is a new handler (or configurable mode) whose output still flows through the existing approval gate — never a bypass of it, and never a replacement for the deterministic engines that must remain independently available and explainable regardless of what runs alongside them.

**Multi-node deployment orchestration.** Kubernetes or an equivalent managed-container platform, as detailed in Section 26 — an orchestration-layer change, not an application-architecture change, since the two-process/shared-database shape already fits that model.

## 29. Glossary

**Approval state machine** — the fixed `pending_review → approved/rejected → optionally rolled_back` status flow shared by every pipeline stage that changes or groups customer data (`CleaningRun`, `StandardizationRun`, `MatchRun`, `ExportRun`).

**Composite tenant foreign key** — the `(organization_id, X) → parent(organization_id, id)` foreign key pattern used by every tenant-scoped table, requiring an explicit `UniqueConstraint(organization_id, id)` on the referenced parent.

**CredentialProvider** — the abstraction (`app/worker/credentials.py`) through which handlers and the execution engine obtain live credentials for a `DataSource`, without knowing or caring where those credentials are actually stored.

**ExecutionContext** — the immutable dataclass (`app/worker/handlers/base.py`) passed to every `ExecutionHandler.execute()` call, carrying the task run, task, data source, idempotency key, and credential provider.

**ExecutionHandler** — the `Protocol` interface every task-type-specific execution unit implements: a single `execute(context) -> str | None` method, raising `TransientExecutionError` or `PermanentExecutionError` on failure.

**HANDLER_REGISTRY** — the `TaskType → ExecutionHandler` mapping (`app/worker/handlers/__init__.py`) that is the sole extension point for adding real behavior for a new task type.

**Idempotency key** — a UUID generated once per `TaskRun` at creation and never changed, including across retries, that handlers must pass to any downstream write so a retried execution cannot create a duplicate downstream effect.

**Lease token** — a fresh, single-use UUID generated on every claim of a `TaskRun`, required (along with `status='running'`) on every subsequent heartbeat and completion call for that claim to have any effect. The core mechanism of concurrency-safe execution ownership (Section 12).

**Permanent failure / PermanentExecutionError** — a handler failure that will never succeed on retry; the engine terminates the run as `failed` immediately regardless of remaining attempts.

**Reaper** — the worker-process loop (`app/worker/reaper.py`) that recovers any `running` `TaskRun` whose lease expired without a heartbeat, reclaiming it exactly as a worker-reported failure would.

**Scheduler** — the worker-process pass (`app/worker/scheduler.py`, Module 12) that atomically creates exactly one `pending` `TaskRun` for each due, active, scheduled `SYNC` task and advances its `next_run_at`.

**Tenant** — an `Organization`; every domain resource belongs to exactly one, and cross-tenant access is uniformly a `404`.

**Transient failure / TransientExecutionError** — a handler failure expected to succeed on retry; the engine requeues the run if attempts remain.

**TaskRun** — the row representing one execution attempt (and its retries) of a `Task`; the unit the execution engine claims, executes, and completes.
