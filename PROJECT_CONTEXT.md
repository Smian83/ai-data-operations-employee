## Project Overview

A multi-tenant SaaS backend that lets an organization register data sources, define tasks against them, and have those tasks executed automatically by a background worker pool — CSV ingestion and profiling, followed by deterministic, human-approved data cleaning. Every resource is tenant-scoped from the database up: two organizations never share data, and that isolation is enforced structurally (composite foreign keys, partial indexes, per-tenant file paths), not just by application-layer checks.

## Current Architecture

- A FastAPI application (`backend/app/main.py`) serving the REST API, and a separate worker process (`python -m app.worker`) that claims and executes `TaskRun`s — two independently deployable, independently scalable processes sharing one PostgreSQL database.
- Multi-tenancy: every domain table carries `organization_id`; tenant-scoped foreign keys use the `(organization_id, X) → parent(organization_id, id)` composite pattern, requiring an explicit `UNIQUE(organization_id, id)` on the referenced parent. Cross-org access is uniformly a 404, indistinguishable from "doesn't exist."
- Execution engine (Module 4): `TaskRun`s are claimed via `SELECT ... FOR UPDATE SKIP LOCKED` (PostgreSQL only), owned via a fresh `lease_token` fencing UUID per claim, kept alive by a heartbeat thread, and recovered by a reaper on lease expiry. A `HANDLER_REGISTRY` maps `TaskType → ExecutionHandler`; adding real behavior for a task type is a new handler module plus one registry line, never a change to the engine itself.
- Migrations are hand-written Alembic (never autogenerated), one linear chain, currently 12 revisions ending at `d5e6f7a8b9c0` (adds `tasks.schedule_interval_seconds`/`tasks.next_run_at`, two `CHECK` constraints, one partial index — see Module 12 below).
- Soft delete via `is_active` + a partial unique index on `lower(trim(name))` scoped to active rows, everywhere a named resource can be deactivated.

## Current Module

**Module 13 (Output Artifact Retention), `module-13-output-artifact-retention` branch — Phases 1 through 5 complete, not yet merged.** Directly closes the storage-retention gap Modules 6/7/9 each left open in their own Known Limitations (rolled-back/rejected/approved runs previously kept output files on disk forever) — once an operator explicitly opts in. Disabled by default (`OUTPUT_RETENTION_ENABLED=false`); nothing is ever evaluated for deletion, and no new query is even opened for this purpose, until that is changed.

Four-state artifact lifecycle, only one state ever persisted: **ACTIVE** (`pending_review`, or terminal but still inside the configured window), **EXPIRED** (terminal and past the window — derived at query time from the run's own status/decision-timestamp columns, never stored), **PURGE_PENDING** (a `SELECT ... FOR UPDATE`/`SKIP LOCKED` row lock held for one artifact's own transaction, never a persisted claim), **PURGED** (`output_deleted_at` is set — the one new durable fact). `app/worker/retention.py::purge_expired_artifacts()` is a bounded pass over all three output-producing run types (cleaning, standardization, export), reusing the identical `SELECT ... FOR UPDATE SKIP LOCKED` + guarded-`UPDATE` concurrency pattern Modules 4/12 already proved under real PostgreSQL concurrency, and the identical one-committed-transaction-per-artifact fault-isolation shape Module 12's scheduler established. `app/artifacts/storage.py`'s new `ArtifactStorage` protocol (`LocalFilesystemArtifactStorage` the sole implementation) is the one place any artifact file is actually opened or deleted; `resolve_artifact_path` (Module 10) remains the sole path-containment boundary, unchanged. Integrated into the worker loop (`run_forever`) as a third independent timer alongside Module 12's scheduler timer and the reaper timer, gated by `RETENTION_POLL_INTERVAL_SECONDS` (default `3600`); a retention-pass failure is caught, rolled back, and logged, never crashing the worker process. All three artifact-download endpoints (Module 10) now return `410 Gone` for a purged artifact, checked before any path resolution or file open, with one audit row still written per attempt (`ArtifactDownloadEvent.outcome="purged"`, no `failure_reason_code`).

Ten new Prometheus metrics (`app/worker/metrics.py`'s Module 13 block) — `retention_passes_total`, `retention_artifacts_eligible_total`, `retention_artifacts_purged_total`, `retention_artifacts_already_missing_total`, `retention_purge_failures_total`, `retention_dry_run_artifacts_total`, `retention_bytes_reclaimed_total`, `retention_pass_duration_seconds`, `retention_oldest_eligible_artifact_age_seconds`, `retention_backlog_artifacts` — all incremented/set exactly once per pass, in bulk, from the pass's own already-computed final tallies (never inside the per-artifact claim loop), making double-counting structurally impossible. The two backlog gauges are computed via plain, lock-free `COUNT`/`MIN` reads, only on the *enabled* path, so the documented "disabled pass opens no query" guarantee is never broken; while disabled, they hold their last real value rather than falsely reporting an empty backlog.

Key decisions made in Module 13 so far (grounded in the approved design, not invented): a `POST /internal/retention/run` manual-trigger endpoint was raised as an open design question ("recommended default: include it, but this is a scope call") but was never one of the three formally approved revisions (Artifact Lifecycle, Storage Abstraction, Success Metrics) — confirmed absent from `app/api/internal.py` and deliberately not built; retention only ever runs on its worker-loop timer. `RETENTION_CLAIM_BATCH_SIZE` is a single, shared, whole-pass budget across all three run types combined, not a separate allowance per type — a production-review finding (the original implementation gave each run type its own fresh batch-size allowance, allowing up to 3x the configured limit in one pass) fixed before this module's Phase 3 was approved. `already_missing` is deliberately not a failure outcome (contrast `ArtifactDownloadEvent.file_missing`, which is) — an artifact already gone still reaches `PURGED` successfully; nothing went wrong. Dry-run mode (`OUTPUT_RETENTION_DRY_RUN=true`) evaluates and audits every eligible artifact exactly as a real pass would but never deletes a file or sets `output_deleted_at`, and may re-discover the same artifact on a later pass without limit, since nothing durable changed. The retention window has a fixed, non-configurable, hand-kept-in-sync-with-the-database-`CHECK`-constraint floor of 7 days (`RETENTION_WINDOW_HARD_FLOOR_DAYS` in `app/core/config.py`, `RETENTION_WINDOW_HARD_FLOOR_DAYS_DB` in `app/models/artifact_retention_event.py`) — the same "two constants, not one derived source of truth" relationship Module 12 already established for its own schedule-interval floor.

Key decisions made in Module 12 (grounded in existing code, not invented): scheduling is `TaskType.SYNC`-only in V1 — TRANSFORM/STANDARDIZE/MATCH/EXPORT/OTHER all require an explicit `source_task_run_id` chaining decision that automatic scheduling alone can't resolve without either weakening the existing approval-gate architecture or guessing which prior run to build on, so they're excluded, not merely deferred by oversight. The scheduler's transaction design was corrected during architectural review from an initial whole-batch-transaction proposal to a bounded loop of independent, single-task transactions specifically for fault isolation (one bad task's rollback never touches another already-committed task's progress in the same pass) and minimal lock duration, without weakening the duplicate-prevention guarantee at all. `triggered_by IS NULL` was kept (not replaced by a new `trigger_type` column) as the sole signal distinguishing scheduler- from manually-created runs, after confirming by direct search that no endpoint anywhere in this codebase hard-deletes a `User` row — the theoretical `ON DELETE SET NULL` ambiguity this raises is real at the schema level but unreachable through the application today. `scheduled_for`/`schedule_occurrence_key` columns were considered and rejected as unnecessary given the per-task transactional guarantee already providing exactly-once creation without them. Duplicate-run prevention under concurrent scheduler workers is proven against genuinely concurrent real-PostgreSQL sessions (two real threads, independent sessions, a `threading.Barrier`), not inferred from SQLite, which has no real row locking. A real bug was found and fixed while authoring the migration, not after: SQLite's `op.batch_alter_table` (required to add `CHECK` constraints to an existing table on that dialect) recreates the whole `tasks` table via a reflect-and-copy step that cannot reflect `ix_tasks_org_name_active` (an expression-based partial index from Module 3) — this silently dropped that index; the migration now explicitly drops and recreates it around each `batch_alter_table` block, on both dialects, guarded by a dedicated regression test.

Key decisions made in Module 6 (grounded in existing code, not invented): reused `TaskType.TRANSFORM` rather than adding a new enum value (already reserved for this per the handler registry's own docstring); `cleaning_runs.status` is a plain string, matching `TaskRunEvent.event_type`'s existing precedent, not a new native Postgres enum; human approval is required for every cleaning run by default (auto-approval exists as a disabled lever, not an assumption). Two real implementation bugs were caught and fixed during integration testing, not present in the design: `CleaningRun` was missing `UniqueConstraint(organization_id, id)` (the same composite-FK-target pattern every other parent table requires — `CleaningChange`'s FK to it failed on SQLite without it), and `CleaningHandler` originally referenced `cleaning_run.id` before the flush that populates it (a Python-side `default=uuid.uuid4` column default). Both are fixed in the merged code. `tests/conftest.py`'s shared per-test table-cleanup fixture was also fixed: Module 6's self-referential `task_runs.source_task_run_id` FK (`RESTRICT`) can violate a single bulk `DELETE` depending on row order, so self-referencing columns are now nulled out before the delete pass, independent of order — a fix to shared test infrastructure, not to Module 6 itself.

Key decisions made in Module 7 (grounded in existing code, not invented): `TaskType.STANDARDIZE` is a genuinely new enum value, not a reuse — unlike Module 6, no existing `TaskType` was reserved for this purpose, so extending the native enum (rather than overloading `EXPORT`/`OTHER`) was the only option that doesn't make one task type mean two unrelated things; `standardization_runs.status`/`standardization_column_mappings.field_type`/`standardization_lookup_entries.field_type` are plain strings, same "small internal/config-owned value set → plain string, validated at the API layer" precedent as `TaskRunEvent.event_type` and `cleaning_runs.status`; the two new organization-configuration tables use two partial unique indexes each (not one) because their uniqueness key includes a nullable column (`data_source_id`, `field_type`) and `NULL != NULL` under standard SQL semantics would otherwise let duplicate NULL-scoped rows through undetected. `StandardizationConfig`'s five scalar settings (default country/currency, date/boolean output form, numeric locale) have no persistence or API surface yet — every run uses all-default config, a documented known limitation, not a silent gap (see Known Limitations). Real-PostgreSQL verification (Phase 5) caught one real bug SQLite verification could not: an auto-derived foreign-key constraint name on `standardization_column_mappings` exceeded PostgreSQL's 63-byte identifier limit — see Coding Standards and Testing Standards below for the resulting lessons.

Key decisions made in Module 8 (grounded in existing code, not invented): `TaskType.MATCH` is a genuinely new enum value for the same reason `STANDARDIZE` was — no existing `TaskType` was reserved for matching, so extending the native enum a second time (`ALTER TYPE task_type_enum ADD VALUE 'match'`) was the only non-overloading option, and this time the migration author had a proven, already-exercised pattern to follow rather than a first attempt. `MatchHandler` writes no output file at all — a deliberate departure from Modules 6/7's shape, not an oversight: matching only groups existing standardized records, so producing a rewritten CSV would mean physically merging/dropping rows, exactly the automatic destructive action the design forbids; Module 8's actual deliverable is the audit data (`MatchRun`/`MatchGroup`/`MatchDecision`/`MatchSkippedBlock` rows). Two design-review additions closed observability gaps before implementation began: `MatchDecision.blocking_key` (nullable — set for every Stage-2 decision, null for Stage-1 exact-row decisions, which use no blocking) makes "why were these two records compared" independently auditable without re-running the engine; the new `match_skipped_blocks` table (one row per block skipped for exceeding `MATCH_MAX_BLOCK_SIZE`, storing the true block size plus a small deterministic sample of row indices, capped by a new `MATCH_MAX_SKIPPED_ROW_SAMPLE` setting) answers the mirror-image "why were these two records never compared" without any unbounded storage growth. `match_rule_sets` reuses the exact two-partial-unique-index pattern Module 7's configuration tables established (nullable `data_source_id` scope, `NULL != NULL` semantics) rather than inventing a new configuration shape. Canonical-record selection is a fixed, deterministic rule (lowest `row_index`) and blocking uses a single highest-weight field only — both explicit, named scope limits carried into this file's Known Limitations below, not silent gaps. Unlike Module 7, Module 8's real-PostgreSQL verification (Phase 5) found **zero** new bugs — every constraint/index identifier was pre-measured against the 63-byte limit during design and implementation (longest is 48 characters), directly applying the lesson Module 7's verification had to discover the hard way.

Key decisions made in Module 9 (grounded in existing code, not invented): `TaskType.EXPORT` required no enum change at all -- it already existed in the base enum (predating even Module 7's first `ALTER TYPE` addition) and had simply been reserved on `NoOpHandler` since Module 4, the first time any module has activated a pre-existing `TaskType` rather than reuse one for an overlapping purpose (Module 6) or add a genuinely new one (Modules 7/8). An approved `MatchRun` is the required upstream gate, mirroring every prior stage's "requires the immediately-prior stage's approved run" pattern. Materialization is deterministic and pure (`app/export/engine.py`): duplicate groups' canonical rows and every ungrouped row survive, non-canonical duplicate rows are excluded, original surviving-row order is preserved -- no field-level merging or enrichment across duplicate records, a named non-goal for this release, not a gap. Two required post-review clarifications materially shaped the final design: (1) the two reserved provenance columns were renamed from unprefixed `canonical_record`/`source_row_index` to namespaced `__aiops_canonical_record`/`__aiops_source_row_index`, with a hard, permanent-failure collision policy (checked before any file write or `ExportRun` persistence, never a silent rename/suffix/overwrite) protecting any customer column of the same name; (2) `export_timestamp` was explicitly excluded from the serialized CSV and confirmed database-metadata-only, so that two independent `EXPORT` `TaskRun`s over identical approved input produce byte-identical files (same `output_sha256`/`output_file_size_bytes`) -- a stronger determinism guarantee than the project's usual task-level idempotency alone. `ExportRun` gained four file-self-description fields (`output_file_size_bytes`, `output_column_count`, `export_timestamp`, `csv_format_version`) so an exported artifact's expected structure can be inspected without reopening the file, with the explicit caveat that actual content-integrity verification against `output_sha256` still requires reading the bytes -- metadata describes the artifact, it doesn't substitute for reading it. `ExportRowExclusion` is bounded per run (`EXPORT_MAX_PERSISTED_EXCLUSIONS`), same aggregate-always-exact/detail-capped pattern as every prior audit table, and is the first table in this project with a composite foreign key into a non-immediate-parent module's table (`match_groups`, owned by Module 8). Real-PostgreSQL verification found zero bugs, and went beyond migration-cycle/test-suite checks for the first time by explicitly attempting seven distinct constraint violations (invalid status, invalid `csv_format_version`, negative `output_file_size_bytes`, the row-count invariant, two different cross-organization composite-FK reference attempts) directly against real Postgres, confirming each was rejected at the database layer, not just the application layer.

Key decisions made in Module 10 (grounded in existing code, not invented): the design's original single-pass "hash while streaming" approach was rejected during architectural review specifically because corrupted or tampered bytes could reach the client before a mismatch was detected -- the corrected design re-verifies the COMPLETE artifact via bounded chunked reads through a single `open()` call before any response body begins, accepting the extra full-file read pass as a deliberate, acceptable tradeoff given this system's existing ~25MB `CSV_MAX_FILE_SIZE_BYTES` ceiling. `ArtifactDownloadEvent.outcome` is a small, explicit, database-enforced controlled set (`started`/`completed`/`integrity_failed`/`file_missing`/`stream_failed`), not a single point-in-time "downloaded" flag -- and deliberately has no separate client-disconnect value, since this stack's FastAPI/Starlette streaming-response lifecycle was never confirmed to reliably distinguish a client disconnect from a genuine server-side I/O failure; `stream_failed` covers both rather than asserting an unverified guarantee. `output_file_path` removal from the three `*Read` schemas was made only after a repository-wide grep for actual consumers, not by default -- Option A (immediate removal) was chosen specifically because the evidence showed the risk was narrow and fully contained (three test assertions, all in Module 9's own suite, all updated as part of this module's own work). The audit row's terminal-outcome finalization deliberately uses its own independent database session rather than the request's injected one, because the success/failure path that matters most (finalizing after a streamed transfer) runs from inside a `StreamingResponse` generator, which the ASGI server drives after the endpoint function has already returned -- by then the request's own session may already be closed. Final review (before merge) found and fixed one real defect: an unguarded `os.fstat()` call between successful verification and response construction that could leak the already-open file descriptor and leave an audit row stuck at `started` forever if it failed -- every other failure branch already closed the descriptor and reached a terminal outcome; this one now does too. This is the first module in the project's history to remove a field from an existing API response schema as part of its own release, rather than only ever adding fields.

Key decisions made in Module 11 (grounded in existing code, not invented, and refined through three design revisions before implementation began): the design went through Discovery → Revision 1 → Revision 2 → Revision 3, each revision driven by specific architectural review corrections, not open-ended iteration -- Revision 1 corrected item-classification and search-capability gaps; Revision 2 removed a proposed `priority` field down to "reserved but always NULL"; Revision 3 removed `priority` a second, final time, all the way to non-existent, on the grounds that an inert always-NULL field and a sort option that silently falls back to another sort are both worse than not shipping the field at all until real prioritization behavior exists to back it. `source` (identifying an item's fine-grained origin, e.g. `cleaning_run`/`match_decision`/`artifact_download_event`) is deliberately a plain, open-ended string, never a closed tuple like `REVIEW_CATEGORIES`/`REVIEW_TYPES` -- the intended pattern for any future module joining this same queue is to add one more `UNION ALL` branch plus one new `source` value, never a runtime write API into a shared table. Two factual corrections to the design document itself were caught only by reading the actual model files during implementation, not before: `match_decisions.reason` is `NOT NULL` and already populated by the matching engine (the design had assumed it didn't exist and should be left NULL), and the design's own "eleven-column shape" language undercounted its own explicitly-retained `label` field by one. Both are documented as implementation-level corrections, not silent deviations. `ExportRun` has no `confidence_score` column (Module 9's row-level deduplication has no scoring concept) -- handled with the same `hasattr`-guarded typed-NULL substitution already applied to the `reason` column on the four run-type branches, rather than inventing a value. Index additions were evidence-gated, not assumed: `EXPLAIN (ANALYZE, BUFFERS)` at realistic near-term scale showed no measurable benefit from any of the six candidate composite indexes, and only a deliberately-seeded ~20,750-row large-tenant scale surfaced a real Seq-Scan-to-Bitmap-Heap-Scan improvement -- a high-selectivity control query against the same large dataset confirmed the same index is correctly left unused where it wouldn't help, satisfying this project's standing "avoid redundant indexes" bar with actual measurement rather than assertion. An independent post-implementation production review (not the same pass that wrote the code) is a new practice introduced by this module -- it found one real, previously-undiagnosed correctness gap (LIMIT/OFFSET pagination was not guaranteed stable when two rows shared an identical `created_at`, a genuine risk given PostgreSQL's `now()` is the transaction timestamp and can be identical across rows written in the same transaction) plus several correct-but-untested edge cases, all resolved and re-verified before a second, final review approved the branch — a two-review-pass pattern worth reusing for any future module with nontrivial query logic.

## Coding Standards

- SQLAlchemy 2.0 typed declarative style (`Mapped[...]`/`mapped_column`) throughout — no legacy `Column`-only style.
- Any native PostgreSQL enum **must** use `sqlalchemy.dialects.postgresql.ENUM(..., create_type=False)`, never the generic `sqlalchemy.Enum` — the generic class silently discards `create_type`, which caused a real, three-round-deep production migration bug (`DuplicateObject`) in Module 4. Alembic migrations are the sole owner of `CREATE TYPE`/`DROP TYPE`; models must never independently emit it.
- Small, worker-owned state fields (like `TaskRunEvent.event_type`) are plain `String`, not a dedicated Postgres enum, when the value set is simple and internal — reserves native enums for values with real external/API meaning.
- Every tenant-scoped table: `organization_id` column, composite FK `(organization_id, X) → parent(organization_id, id)`, and where referenced as a composite-FK target, an explicit `UniqueConstraint(organization_id, id)`. This is easy to miss on a brand-new parent table (Module 6's `CleaningRun` shipped without it initially and only failed once `CleaningChange`'s composite FK to it was actually exercised against SQLite) — add it at the same time the table is created, not after the first FK-target failure surfaces.
- Soft delete via `is_active`, never a hard `DELETE`, for anything another row might reference.
- Docstrings explain *why*, not just *what* — this is a deliberate, consistent convention across every module so far; new code should match it, not drop it for terseness.
- Pydantic v2 schemas use `ConfigDict(from_attributes=True)`; response models never expose internal worker-ownership fields (e.g. `lease_token` is deliberately absent from `TaskRunRead`).
- PostgreSQL identifier names (table, column, constraint, index) have a hard 63-byte limit (`NAMEDATALEN`), silently unenforced by SQLite. This project's `fk_{table}_{column}_{ref_table}` hand-written naming convention is safe for short table names but not guaranteed as table names grow — Module 7's `standardization_column_mappings`/`standardization_lookup_entries` (the longest table names yet) pushed their `organization_id` FK name to 64 and exactly 63 characters respectively, only caught by real-Postgres verification (Phase 5). When a hand-written constraint name is derived from a long table name, check its length before it ships, or use this project's existing shorter `_org_`-style abbreviation for that column outright.
- Extending an *existing* native PostgreSQL enum (`ALTER TYPE ... ADD VALUE`) is a different migration shape from creating a new enum type (every prior enum-touching migration in this project did the latter): the new label must be added inside Alembic's `autocommit_block()` (PostgreSQL disallows using a newly added enum value within the same transaction it was added in), and there is no `DROP VALUE` — a downgrade leaves the label in place, which is the standard, expected, non-reversible-in-that-one-respect behavior, not a bug to work around. Module 8 repeated this exact shape for `ALTER TYPE task_type_enum ADD VALUE 'match'` with zero new issues, confirming it's now a proven, reusable pattern rather than a one-off.
- **Lesson applied proactively, not discovered the hard way a second time:** every hand-written constraint/index name in Module 8's migration was measured against PostgreSQL's 63-byte `NAMEDATALEN` limit *before* the migration was written, directly because Module 7's real-Postgres verification had to catch this after the fact. Module 8's longest identifier is 48 characters; real-PostgreSQL verification (Phase 5) found zero identifier-length issues, the first module where this class of bug didn't need catching at all.
- **Deterministic blocking, not full pairwise comparison, is this project's pattern for bounding any O(n²)-shaped matching/comparison workload.** Module 8's matching engine (`app/matching/blocking.py`) buckets candidate rows by a single, highest-weight configured field's normalized value before ever comparing two rows — comparison cost becomes the sum of each block's `size²`, not the whole dataset's, and a block exceeding a configured cap (`MATCH_MAX_BLOCK_SIZE`) is skipped entirely rather than silently truncated or allowed to blow up compute. Any future feature comparing many records pairwise should reach for this same blocking-before-comparing shape rather than a raw nested loop.
- **Bounded, deterministic detail sampling is the project's answer to "the aggregate must stay cheap to compute, but some inspectable detail should survive even when the full detail set doesn't."** Module 8's `MatchSkippedBlock.sample_row_indices` (capped by `MATCH_MAX_SKIPPED_ROW_SAMPLE`, always the deterministic *lowest* row indices, never random) is the reference implementation: the aggregate count (`MatchRun.skipped_block_count`) is always exact, while the per-row detail is capped on both axes (row count per run structurally bounded by `row_count / MATCH_MAX_BLOCK_SIZE`, and each row's own sample array separately capped). Prefer this two-tier shape — exact aggregate, capped-and-deterministic detail — over either an unbounded detail table or a detail table with no aggregate at all.
- Partial unique indexes for a nullable-scope configuration table (Module 7's `standardization_column_mappings`/`standardization_lookup_entries` pattern: two partial indexes, not one, because `NULL != NULL`) is now a twice-reused pattern — Module 8's `match_rule_sets` (nullable `data_source_id` scope) applied it a second time with no modification needed. Reach for this exact shape, not a new one, whenever a configuration table's uniqueness key includes an "applies to everything" nullable column.

- **Reserved system columns must use a namespace and fail safely on collision.** Module 9's provenance columns are `__aiops_canonical_record`/`__aiops_source_row_index`, not the unprefixed names originally proposed -- caught during architectural review before implementation. Any future module that injects its own metadata columns into a customer-facing artifact should reserve a namespaced prefix up front and treat a collision with an existing customer column as a hard, permanent failure (checked before any write or persistence, never silently renamed, suffixed, or overwritten), not something discovered after the fact.
- **Run-specific timestamps must stay out of deterministic output artifacts.** `ExportRun.export_timestamp` is database metadata only, deliberately never serialized into the exported CSV -- the general pattern for any future module producing a byte-comparable output file: persist wall-clock/run-identifying data on the database row, never inside the artifact whose determinism you want to guarantee.
- **File metadata describes expected structure; hash verification still requires reading bytes.** `ExportRun`'s four self-description fields (`output_file_size_bytes`, `output_column_count`, `export_timestamp`, `csv_format_version`) let a caller inspect an artifact's expected shape without parsing it, but this does not substitute for actually reading the file when true content-integrity verification against `output_sha256` is required -- state this distinction explicitly in any future module with similar metadata, rather than letting "inspect without reopening" be read as "verify without reopening."
- **Byte-identical fresh exports are a stronger determinism guarantee than task-level idempotency alone.** This project's standing idempotency contract (retry the same `TaskRun`, get the same row, never a duplicate) is necessary but not sufficient for an artifact-producing module -- Module 9 also proves that two *independent* `TaskRun`s over identical approved input produce byte-identical output files. Any future module producing a deterministic artifact should test both properties separately, not assume one implies the other.
- **Pre-checking PostgreSQL identifier lengths avoids late migration failures.** Module 9's migration had every constraint/index name measured against the 63-byte `NAMEDATALEN` limit before it was written (longest is 58 characters), continuing the discipline Module 8 first applied proactively -- real-PostgreSQL verification (Phase 5) found zero identifier-length issues, the second consecutive module where this class of bug required no after-the-fact catching.
- **Real-PostgreSQL constraint verification should explicitly attempt invalid tenant references and CHECK violations, not just run the test suite.** Module 9's Phase 5 verification went a step further than Modules 7/8's: beyond the full test suite and migration cycle, it directly attempted seven distinct violations against real Postgres (an invalid `status`, an invalid `csv_format_version`, a negative `output_file_size_bytes`, the row-count invariant, and two separate cross-organization composite-FK reference attempts) and confirmed each was rejected at the database layer. This is now the standard extra step for any future module's real-Postgres verification, not just a Module-9-only one-off.

## Testing Standards

- SQLite is the sandbox/CI-speed verification target; it is **not** authoritative for anything PostgreSQL-specific (`FOR UPDATE SKIP LOCKED`, native enum DDL). Real PostgreSQL verification is the final word before any merge.
- Every new API endpoint needs explicit tests for: success path, cross-organization 404, not-found 404, and (where applicable) invalid-state-transition responses — no endpoint ships without this, following the standard set (and initially missed, then corrected in 5.1) by the profile endpoint.
- Every new worker handler needs an idempotency test proving a retried run doesn't duplicate its side effect (unique-constraint-plus-refetch pattern).
- Any handler touching tenant-supplied file paths needs an explicit cross-tenant-isolation test, not just a traversal-escape test — the two are different failure modes, both required (established by the B1 fix).
- A self-referential FK on a table (e.g. `task_runs.source_task_run_id → task_runs.id`, `RESTRICT`) needs the shared test-cleanup fixture to null out self-referencing columns before its bulk-delete pass — a plain per-table `DELETE` can otherwise violate `RESTRICT` depending on row order within that single statement (established by the Module 6 `_clean_tables` fix in `tests/conftest.py`).
- SQLite enforces neither PostgreSQL's 63-byte identifier length limit nor, independently, the `CHECK` constraint a migration-local native-enum column declares unless that migration explicitly sets `create_constraint=True` (this project's migrations never do — only the ORM model's own copy of the enum object does, and Alembic migrations never consult the ORM model). Both were confirmed empirically against a real Alembic-migrated SQLite database during Module 7, not assumed. Neither class of bug is catchable on SQLite alone — real-PostgreSQL verification (Phase 5) is what actually confirms identifier-length and enum-constraint correctness, not just migration-cycle symmetry.
- `tests/conftest.py`'s `_create_tables` fixture builds tables via `Base.metadata.create_all()`, not Alembic — against real PostgreSQL this requires the native enum *types* to already exist first (`create_type=False` on the ORM models means `create_all()` never creates them itself), so `alembic upgrade head` must be run once against the target Postgres database before the first test run against it. Re-running it on a later session is safe (a no-op once already at head) — documented in the README's "Running the same suite against real PostgreSQL" section, re-confirmed during Module 7's Phase 5 verification.
- Current baseline: 580 tests passing (526 baseline + 54 added in Module 12), full suite re-run required before every merge.
- **A pure, framework-independent query/aggregation function is independently unit-testable without a running API or TestClient.** Module 11's `fetch_review_queue()` (`app/review_queue/query.py`) takes a `Session` and plain dataclasses in, returns a plain dataclass out — its 22 unit tests (`tests/test_review_queue_query.py`) call it directly, with a handful of API-level tests (`tests/test_review_queue_api.py`) reserved only for what's genuinely HTTP-boundary-specific (auth, real end-to-end worker-produced data, query-param validation, response-shape). Any future module built as a pure aggregation/query layer over existing tables should follow this same split rather than testing everything through the API.
- **Deterministic pagination requires a secondary sort key whenever the primary key can plausibly tie.** PostgreSQL's `now()` is the *transaction* timestamp, constant across every statement in one transaction — rows written together (a common real pattern) can share an identical `created_at`. Module 11's post-merge-review fix (`ORDER BY <primary>, reference_id ASC`) is the standing pattern: any future endpoint sorting by a timestamp or a score column under `LIMIT`/`OFFSET` needs a stable, always-unique secondary key, not just a primary sort column, or repeated/paginated requests are not guaranteed to return a stable result set.
- **An independent, adversarial-minded review pass after implementation (not just before it) catches gaps a same-pass author won't.** Module 11 introduced a second, separate production review after the implementation review that approved the initial code — it found the pagination-determinism gap above and several correct-but-untested edge cases (whitespace-only search, malformed-UUID search, combined multi-dimension filters) that the first review's own checklist had already named but the initial test suite hadn't actually covered. Worth reusing for any future module with nontrivial query/filter logic, not just a one-off for this module.

## Production Review Checklist

Applied to every module before merge (the exact ten dimensions used for Module 5's review, now the standing bar):

1. Security (tenant isolation, credential handling, file/path safety)
2. Migration correctness (chain integrity, single head, up/down symmetry)
3. PostgreSQL compatibility (not just SQLite-clean)
4. API consistency (matches existing endpoints' conventions and error shapes)
5. Typing (no untyped/loosely-typed public surfaces)
6. Performance (bounded I/O, no unbounded growth)
7. Edge cases (empty input, zero rows, maximum-size input)

9. Documentation (README + this file kept current)
10. Production readiness (a clear APPROVED/NOT APPROVED verdict with exact required fixes if not approved)

## Security Principles

- Tenant isolation is structural, not incidental: composite FKs, partial unique indexes scoped to active rows, and — as of 5.1 — per-tenant filesystem directories for any tenant-supplied file path. A path is never resolved against a shared root; it's resolved against that tenant's own root, with traversal-escape checks on top, not instead.
- Secrets never live in general-purpose metadata columns — `connection_metadata` is scanned against a secret-key denylist before it's ever persisted. Real credentials live only in `DataSourceCredential`, encrypted at rest (Fernet), accessed only through the `CredentialProvider` abstraction so the execution engine never knows or cares where secrets actually live.
- Concurrency safety under multiple workers is a fencing-token design (`lease_token`, regenerated per claim), not a trust-the-worker-id design — a worker whose lease was reclaimed can never corrupt state it no longer owns, even if it wakes up late.
- Every endpoint sits behind the same authentication dependency; there is no endpoint-specific auth mechanism to keep track of.

## Project Rules

Standing rules for all future work on this project, accumulated from explicit direction given throughout its development:

- Never rebuild, redesign, or restart a completed module without an explicit, specific reason.
- Never guess. If an assumption about existing behavior is uncertain, inspect the actual code before proposing or making a change.
- Never invent architecture that isn't already present — new modules extend existing patterns (handler registry, tenant-scoped FKs, bounded/sampled persistence) rather than introducing parallel ones.
- Never skip testing, and never treat SQLite-only verification as sufficient for a PostgreSQL-targeted change.
- No breaking changes to existing API contracts without an explicit, separate decision to do so.
- Preserve backward compatibility by default; additive changes are the norm.
- Recommend one production-ready approach with justification rather than presenting multiple alternative architectures, unless alternatives are explicitly requested. Note significant trade-offs briefly, then proceed with the recommendation.
- Wait for explicit approval before implementation begins on any new module, and before any merge to `main`.

## Known Limitations

- `SELECT ... FOR UPDATE SKIP LOCKED` claim-concurrency safety is PostgreSQL-only and cannot be exercised on SQLite — sandbox test results for claim-atomicity are indicative, not conclusive; real-Postgres verification is what actually confirms it.
- No in-process cancellation of a running worker handler — `WORKER_DEFAULT_TIMEOUT_SECONDS` governs lease-based reclaim by the reaper, not interruption of work already in progress. A pathologically slow handler run isn't stopped mid-flight, only eventually reclaimed.
- CSV is the only supported source type for real execution (`SourceType.POSTGRES`/`MYSQL`/`REST_API`/`S3`/`OTHER` all fail permanently under `SYNC`); connectors for other source types are unbuilt.
- CSV files are read entirely into memory, bounded by `CSV_MAX_FILE_SIZE_BYTES` (25 MB default) — no streaming path for larger files.
- Sandbox test/verification runs on Python 3.10; the production Docker image targets Python 3.13. These have not been cross-verified against each other in this project's history — worth confirming before a first real production deploy.
- No AI-assisted or ML-based data correction anywhere in the system yet — Module 6's cleaning engine is deterministic-rules-only by design, with AI-assisted correction explicitly deferred as a defined extension point, not built.
- Module 6 rollback is whole-run only (no cell-level selective undo), and rolled-back/rejected cleaning runs leave their output files on disk rather than deleting them — a storage retention policy is deferred, not solved.
- Module 7's `StandardizationConfig` (organization-configured default country/currency, date/boolean output display form, numeric locale) has no database persistence or API surface yet — only the two config tables specified in the approved design (`standardization_column_mappings`, `standardization_lookup_entries`) are implemented and wired. Every standardization run currently uses all-default config, so phone/state-province/postal-code standardization without a resolvable country column, the ambiguous `$` currency symbol, and locale-ambiguous numeric separators are conservatively left untouched rather than guessed.
- Module 7's casing rules are deliberately not locale-aware (no special-casing of "McDonald", "O'Brien", or lower-case connector words) and its postal-address/city rules do no structural parsing, canonical-name lookup, or USPS CASS-style certification — light-touch format normalization only, a named scope limit from the design, not an oversight.
- Module 7 rollback is whole-run only, same scope limit as Module 6, and likewise never deletes the standardized output file on rollback/rejection.
- Module 8 performs no physical merge, export, or deletion of duplicate records anywhere — approving a `MatchRun` is a pure status transition confirming the discovered groupings are correct, never an automatic action on the underlying data. Acting on those groupings (producing an actual deduplicated dataset) is a deliberately separate, unbuilt future capability, gated on the still-unbuilt `EXPORT` handler and its own future design/approval step.
- Module 8's canonical-record selection is fixed (lowest `row_index` / first-seen) and not organization-configurable — content-aware strategies ("most complete row," "most recently updated") are a named future extension point, not built.
- Module 8's blocking supports only a single highest-weight field as the blocking key; multi-field/composite blocking (which could catch more true duplicates that happen to disagree on the single highest-weight field) is a deferred accuracy improvement.
- Module 8 does no fuzzy, phonetic, or approximate matching of any kind (`comparison_type` is limited to `exact`/`normalized_exact`) — a hard, deliberate scope limit for explainability and reproducibility, not an oversight, and going further than the project's general "no AI/ML" rule by excluding classical approximate-matching algorithms too.
- Module 8's `match_rule_sets`/`match_rule_fields` configuration is immutable and versioned with no partial-update endpoint — changing even one field's weight requires creating an entirely new rule set (and re-running matching to see its effect); a future "clone and adjust" convenience endpoint is not precluded but isn't built.
- Module 8 matches exactly one `StandardizationRun` per `MatchRun` (no cross-run matching across multiple standardization runs/tasks/data sources within a single match run), and rollback is whole-run only, same scope limit as Modules 6/7.

- Module 9 performs row-level deduplication only, never field-level merging or enrichment across duplicate records -- the canonical row wins entirely, exactly as Module 8 selected it. Combining values from non-canonical duplicates into the canonical record is a materially different capability, explicitly deferred, not solved here.
- Module 9 has no organization-configurable export behavior in this release -- one fixed materialization strategy, no choice of output format, no option to retain excluded rows with a flag instead of dropping them, no configurable column selection. The two provenance columns are fixed, reserved, and non-configurable by design.
- Module 9 has no delivery mechanism -- output lands in tenant-scoped local storage (`CSV_EXPORTED_ROOT`) only, same as every prior output-producing module. Emailing, uploading, or otherwise delivering the exported file is out of scope.
- Module 9 rollback never deletes the physical export file, extending the same retention-policy gap already carried from Modules 6/7 to a third output-producing module.
- Module 9 matches exactly one `MatchRun` per `ExportRun` (no combining multiple match runs, data sources, or organizations into one export), same one-upstream-run-per-run scope limit Module 8 established for its own `StandardizationRun` relationship.
- If a standardized dataset already contains a column literally named `__aiops_canonical_record` or `__aiops_source_row_index`, Module 9 cannot export it as-is -- the customer would need to rename that source-side column upstream first. No in-module workaround (auto-rename/suffix) is offered, by design.

- Module 10 supports no HTTP range requests -- a download always transfers the complete artifact in one request; there is no way to resume a partial transfer or fetch a byte range.
- Module 10 exposes no customer-facing download-history endpoint -- `artifact_download_events` exists for audit/compliance purposes only in this release; nothing in the API lets a caller list their own past download attempts.
- Module 10 has no dedicated rate limiting on the download endpoints, consistent with every other endpoint in this project (a project-wide gap, not unique to this module).
- Module 10's `stream_failed` outcome does not distinguish a client-initiated disconnect from a genuine server-side streaming error -- whether this stack's FastAPI/Starlette response lifecycle can reliably make that distinction was never established, so both are recorded identically rather than asserting an unverified guarantee.
- Module 10's `rolled_back`-run downloads are for audit/investigation only and are explicitly NOT current authoritative output -- the `X-Artifact-Run-Status` response header always carries the run's actual status so a client is never left to infer this, but the API itself does not prevent a client from downloading (and potentially misusing) a superseded artifact.
- **Intentional API contract change:** `output_file_path` was removed from `CleaningRunRead`, `StandardizationRunRead`, and `ExportRunRead` in Module 10 -- a client that previously read this field from those three summary endpoints will no longer find it there. The underlying SQLAlchemy model columns and worker-internal handler code are unchanged; only the three API-facing Pydantic `Read` schemas had the field removed. Retrieve the artifact's bytes via the new `GET .../{cleaning,standardization,export}/download` endpoints instead of constructing a local path from the old field.

- Module 11 is strictly read-only V1 by design, not by oversight: no claim, assignment, ownership, locking, notification, escalation-timer, or reviewer-workflow behavior exists anywhere in `GET /review-queue` -- every approve/reject/rollback action still happens through the existing per-run endpoints already in `app/api/tasks.py`. Adding any of these is a genuinely separate, future module, not an extension of this one.
- Module 11 has no `priority` field or prioritization logic of any kind, in any form (not even a reserved, always-NULL placeholder — that was explicitly rejected during design review). Reintroducing prioritization is a future module's decision, made only once real, tested prioritization behavior exists to back it.
- Module 11's search supports exact-UUID match only for Task ID and Run ID (no prefix matching) and case-insensitive substring match only for Task Name and Dataset Name — Artifact Filename search was considered and deliberately left out of V1, since the deterministic filename could not be represented in a query verified identically portable across SQLite and PostgreSQL.
- Module 11's index evidence is fully independently large-scale-tested for only one of the six new indexes (`ix_cleaning_runs_org_status`, tested against a real ~20,750-row seeded table) — the other five are added by structural analogy (identical column shape: a UUID `organization_id` FK plus a small closed-vocabulary status/decision column) and this is disclosed explicitly in both the migration's own docstring and the implementation notes, not asserted as equally verified.
- Module 11's `source` field is deliberately open-ended (a plain string, not a closed vocabulary) specifically so a future module can join the same queue by adding one more `UNION ALL` branch and one new `source` value — but no such branch exists yet beyond the seven authoritative tables Module 11 itself reads from.

- Module 12 schedules `TaskType.SYNC` tasks only — TRANSFORM/STANDARDIZE/MATCH/EXPORT/OTHER cannot be scheduled, by design: those types require an explicit `source_task_run_id` chaining decision that automatic scheduling alone can't resolve without either weakening the existing approval-gate architecture or guessing which prior run to build on.
- Module 12's schedule format is a fixed elapsed-seconds interval only — no cron syntax and no wall-clock/timezone-aligned scheduling (e.g. "daily at 9am in a given timezone") in this release; a documented, non-breaking extension point, not a gap discovered after the fact.
- Module 12 has no task-reactivation-after-soft-delete behavior, because none exists anywhere in the API yet (`TaskUpdate` has no `is_active` field, confirmed by direct inspection during design) — `DELETE /tasks/{id}` remains one-way. If a future module adds reactivation, it must recompute `next_run_at` from the reactivation moment rather than reviving a stale stored value (documented in `docs/module-12-scheduled-task-execution-design.md` Section 6/9).
- The four new Module 12 scheduler metrics (`task_scheduler_passes_total`, `task_scheduler_runs_created_total`, `task_scheduler_errors_total`, `task_scheduler_last_success_timestamp_seconds`) inherit a **pre-existing, project-wide** limitation, not a new one: `/internal/metrics` runs in the FastAPI process and reads that process's own in-memory Prometheus registry, while the worker loop (where these and the four Module 4 execution-engine counters actually increment) runs as a separate OS process with its own independent copy of the same registry object. `tasks_claimed_total`/`tasks_completed_total`/`tasks_failed_total`/`tasks_retried_total` were already unreachable via `/internal/metrics` before Module 12 for the identical reason (only `queue_depth` is genuinely live there, since it's recomputed from a direct DB query per request rather than read from an in-memory counter). Closing this process-boundary gap — e.g. a metrics endpoint inside the worker process, or a push-gateway — is unbuilt, project-wide infrastructure work, not scoped to any one module.

- **Module 13 is not yet merged into `main`** — Phases 1 through 5 (migration/models/config, `ArtifactStorage` abstraction, retention worker + audit + concurrency, worker/API integration + `410`, metrics + documentation) are complete on `module-13-output-artifact-retention`; Phase 6 (final full-suite verification) and explicit merge approval are still pending. Everything else in this Known Limitations block describes the module as implemented on that branch, not as shipped on `main`.
- Module 13 is opt-in and disabled by default (`OUTPUT_RETENTION_ENABLED=false`) — it only closes the storage-retention gap Modules 6/7/9 named in their own Known Limitations once an operator explicitly enables it; a fresh deployment behaves exactly as before Module 13 existed.
- Module 13's retention window is a single, global setting (`OUTPUT_RETENTION_WINDOW_DAYS`) — there is no per-organization or per-run-type override; every tenant and every one of the three run types (cleaning/standardization/export) shares the same configured window and the same fixed 7-day hard floor.
- Module 13 has no manual, on-demand retention-trigger API endpoint — considered during design (`POST /internal/retention/run`) but never approved; retention only ever runs on its worker-loop timer (`RETENTION_POLL_INTERVAL_SECONDS`).
- Module 13's ten new Prometheus metrics inherit the same pre-existing, project-wide worker-process/API-process visibility gap already documented for Module 12's four scheduler metrics above — `/internal/metrics` runs in the FastAPI process and reads that process's own in-memory registry, while these metrics actually increment inside the separate worker process. Closing this is unbuilt, project-wide infrastructure work, not scoped to Module 13.
- Module 13 only deletes the output artifact file itself — `CleaningChange`/`StandardizationChange`/`MatchDecision`/`ExportRowExclusion` and every other audit-table row for a purged run are retained forever, by design; retention is strictly about reclaiming file storage, never about audit-trail pruning.
- Module 13 provides no restore/undelete capability — once `output_deleted_at` is set on a real (non-dry-run) pass, the file is genuinely gone; there is no soft-delete or trash/recovery window.
