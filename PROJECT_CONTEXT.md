## Project Overview

A multi-tenant SaaS backend that lets an organization register data sources, define tasks against them, and have those tasks executed automatically by a background worker pool — CSV ingestion and profiling, followed by deterministic, human-approved data cleaning. Every resource is tenant-scoped from the database up: two organizations never share data, and that isolation is enforced structurally (composite foreign keys, partial indexes, per-tenant file paths), not just by application-layer checks.

## Current Architecture

- A FastAPI application (`backend/app/main.py`) serving the REST API, and a separate worker process (`python -m app.worker`) that claims and executes `TaskRun`s — two independently deployable, independently scalable processes sharing one PostgreSQL database.
- Multi-tenancy: every domain table carries `organization_id`; tenant-scoped foreign keys use the `(organization_id, X) → parent(organization_id, id)` composite pattern, requiring an explicit `UNIQUE(organization_id, id)` on the referenced parent. Cross-org access is uniformly a 404, indistinguishable from "doesn't exist."
- Execution engine (Module 4): `TaskRun`s are claimed via `SELECT ... FOR UPDATE SKIP LOCKED` (PostgreSQL only), owned via a fresh `lease_token` fencing UUID per claim, kept alive by a heartbeat thread, and recovered by a reaper on lease expiry. A `HANDLER_REGISTRY` maps `TaskType → ExecutionHandler`; adding real behavior for a task type is a new handler module plus one registry line, never a change to the engine itself.
- Migrations are hand-written Alembic (never autogenerated), one linear chain, currently 8 revisions ending at `f1a2b3c4d5e6`.
- Soft delete via `is_active` + a partial unique index on `lower(trim(name))` scoped to active rows, everywhere a named resource can be deactivated.

Key decisions made in Module 6 (grounded in existing code, not invented): reused `TaskType.TRANSFORM` rather than adding a new enum value (already reserved for this per the handler registry's own docstring); `cleaning_runs.status` is a plain string, matching `TaskRunEvent.event_type`'s existing precedent, not a new native Postgres enum; human approval is required for every cleaning run by default (auto-approval exists as a disabled lever, not an assumption). Two real implementation bugs were caught and fixed during integration testing, not present in the design: `CleaningRun` was missing `UniqueConstraint(organization_id, id)` (the same composite-FK-target pattern every other parent table requires — `CleaningChange`'s FK to it failed on SQLite without it), and `CleaningHandler` originally referenced `cleaning_run.id` before the flush that populates it (a Python-side `default=uuid.uuid4` column default). Both are fixed in the merged code. `tests/conftest.py`'s shared per-test table-cleanup fixture was also fixed: Module 6's self-referential `task_runs.source_task_run_id` FK (`RESTRICT`) can violate a single bulk `DELETE` depending on row order, so self-referencing columns are now nulled out before the delete pass, independent of order — a fix to shared test infrastructure, not to Module 6 itself.

Key decisions made in Module 7 (grounded in existing code, not invented): `TaskType.STANDARDIZE` is a genuinely new enum value, not a reuse — unlike Module 6, no existing `TaskType` was reserved for this purpose, so extending the native enum (rather than overloading `EXPORT`/`OTHER`) was the only option that doesn't make one task type mean two unrelated things; `standardization_runs.status`/`standardization_column_mappings.field_type`/`standardization_lookup_entries.field_type` are plain strings, same "small internal/config-owned value set → plain string, validated at the API layer" precedent as `TaskRunEvent.event_type` and `cleaning_runs.status`; the two new organization-configuration tables use two partial unique indexes each (not one) because their uniqueness key includes a nullable column (`data_source_id`, `field_type`) and `NULL != NULL` under standard SQL semantics would otherwise let duplicate NULL-scoped rows through undetected. `StandardizationConfig`'s five scalar settings (default country/currency, date/boolean output form, numeric locale) have no persistence or API surface yet — every run uses all-default config, a documented known limitation, not a silent gap (see Known Limitations). Real-PostgreSQL verification (Phase 5) caught one real bug SQLite verification could not: an auto-derived foreign-key constraint name on `standardization_column_mappings` exceeded PostgreSQL's 63-byte identifier limit — see Coding Standards and Testing Standards below for the resulting lessons.

Key decisions made in Module 8 (grounded in existing code, not invented): `TaskType.MATCH` is a genuinely new enum value for the same reason `STANDARDIZE` was — no existing `TaskType` was reserved for matching, so extending the native enum a second time (`ALTER TYPE task_type_enum ADD VALUE 'match'`) was the only non-overloading option, and this time the migration author had a proven, already-exercised pattern to follow rather than a first attempt. `MatchHandler` writes no output file at all — a deliberate departure from Modules 6/7's shape, not an oversight: matching only groups existing standardized records, so producing a rewritten CSV would mean physically merging/dropping rows, exactly the automatic destructive action the design forbids; Module 8's actual deliverable is the audit data (`MatchRun`/`MatchGroup`/`MatchDecision`/`MatchSkippedBlock` rows). Two design-review additions closed observability gaps before implementation began: `MatchDecision.blocking_key` (nullable — set for every Stage-2 decision, null for Stage-1 exact-row decisions, which use no blocking) makes "why were these two records compared" independently auditable without re-running the engine; the new `match_skipped_blocks` table (one row per block skipped for exceeding `MATCH_MAX_BLOCK_SIZE`, storing the true block size plus a small deterministic sample of row indices, capped by a new `MATCH_MAX_SKIPPED_ROW_SAMPLE` setting) answers the mirror-image "why were these two records never compared" without any unbounded storage growth. `match_rule_sets` reuses the exact two-partial-unique-index pattern Module 7's configuration tables established (nullable `data_source_id` scope, `NULL != NULL` semantics) rather than inventing a new configuration shape. Canonical-record selection is a fixed, deterministic rule (lowest `row_index`) and blocking uses a single highest-weight field only — both explicit, named scope limits carried into this file's Known Limitations below, not silent gaps. Unlike Module 7, Module 8's real-PostgreSQL verification (Phase 5) found **zero** new bugs — every constraint/index identifier was pre-measured against the 63-byte limit during design and implementation (longest is 48 characters), directly applying the lesson Module 7's verification had to discover the hard way.

## Coding Standards

- SQLAlchemy 2.0 typed declarative style (`Mapped[...]`/`mapped_column`) throughout — no legacy `Column`-only style.
- Any native PostgreSQL enum **must** use `sqlalchemy.dialects.postgresql.ENUM(..., create_type=False)`, never the generic `sqlalchemy.Enum` — the generic class silently discards `create_type`, which caused a real, three-round-deep production migration bug (`DuplicateObject`) in Module 4. Alembic migrations are the sole owner of `CREATE TYPE`/`DROP TYPE`; models must never independently emit it.
- Small, worker-owned state fields (like `TaskRunEvent.event_type`) are plain `String`, not a dedicated Postgres enum, when the value set is simple and internal — reserves native enums for values with real external/API meaning.
- Every tenant-scoped table: `organization_id` column, composite FK `(organization_id, X) → parent(organization_id, id)`, and where referenced as a composite-FK target, an explicit `UniqueConstraint(organization_id, id)`. This is easy to miss on a brand-new parent table (Module 6's `CleaningRun` shipped without it initially and only failed once `CleaningChange`'s composite FK to it was actually exercised against SQLite) — add it at the same time the table is created, not after the first FK-target failure surfaces.
- Soft delete via `is_active`, never a hard `DELETE`, for anything another row might reference.
- Docstrings explain *why*, not just *what* — this is a deliberate, consistent convention across every module so far; new code should match it, not drop it for terseness.
- Pydantic v2 schemas use `ConfigDict(from_attributes=True)`; response models never expose internal worker-ownership fields (e.g. `lease_token` is deliberately absent from `TaskRunRead`).
- PostgreSQL identifier names (table, column, constraint, index) have a hard 63-byte limit (`NAMEDATALEN`), silently unenforced by SQLite. This project's `fk_{table}_{column}_{ref_table}` hand-written naming convention is safe for short table names but not guaranteed as table names grow — Module 7's `standardization_column_mappings`/`standardization_lookup_entries` (the longest table names yet) pushed their `organization_id` FK name to 64 and exactly 63 characters respectively, only caught by real-Postgres verification (Phase 5). When a hand-written constraint name is derived from a long table name, check its length before it ships, or use this project's existing shorter `_org_`-style abbreviation for that column outright.
- Extending an *existing* native PostgreSQL enum (`ALTER TYPE ... ADD VALUE`) is a different migration shape from creating a new enum type (every prior enum-touching migration in this project did the latter): the new label must be added inside Alembic's `autocommit_block()` (PostgreSQL disallows using a newly added enum value within the same transaction it was added in), and there is no `DROP VALUE` — a downgrade leaves the label in place, which is the standard, expected, non-reversible-in-that-one-respect behavior, not a bug to work around. Module 8 repeated this exact shape for `ALTER TYPE task_type_enum ADD VALUE 'match'` with zero new issues, confirming it's now a proven, reusable pattern rather than a one-off.
- **Lesson applied proactively, not discovered the hard way a second time:** every hand-written constraint/index name in Module 8's migration was measured against PostgreSQL's 63-byte `NAMEDATALEN` limit *before* the migration was written, directly because Module 7's real-Postgres verification had to catch this after the fact. Module 8's longest identifier is 48 characters; real-PostgreSQL verification (Phase 5) found zero identifier-length issues, the first module where this class of bug didn't need catching at all.
- **Deterministic blocking, not full pairwise comparison, is this project's pattern for bounding any O(n²)-shaped matching/comparison workload.** Module 8's matching engine (`app/matching/blocking.py`) buckets candidate rows by a single, highest-weight configured field's normalized value before ever comparing two rows — comparison cost becomes the sum of each block's `size²`, not the whole dataset's, and a block exceeding a configured cap (`MATCH_MAX_BLOCK_SIZE`) is skipped entirely rather than silently truncated or allowed to blow up compute. Any future feature comparing many records pairwise should reach for this same blocking-before-comparing shape rather than a raw nested loop.
- **Bounded, deterministic detail sampling is the project's answer to "the aggregate must stay cheap to compute, but some inspectable detail should survive even when the full detail set doesn't."** Module 8's `MatchSkippedBlock.sample_row_indices` (capped by `MATCH_MAX_SKIPPED_ROW_SAMPLE`, always the deterministic *lowest* row indices, never random) is the reference implementation: the aggregate count (`MatchRun.skipped_block_count`) is always exact, while the per-row detail is capped on both axes (row count per run structurally bounded by `row_count / MATCH_MAX_BLOCK_SIZE`, and each row's own sample array separately capped). Prefer this two-tier shape — exact aggregate, capped-and-deterministic detail — over either an unbounded detail table or a detail table with no aggregate at all.
- Partial unique indexes for a nullable-scope configuration table (Module 7's `standardization_column_mappings`/`standardization_lookup_entries` pattern: two partial indexes, not one, because `NULL != NULL`) is now a twice-reused pattern — Module 8's `match_rule_sets` (nullable `data_source_id` scope) applied it a second time with no modification needed. Reach for this exact shape, not a new one, whenever a configuration table's uniqueness key includes an "applies to everything" nullable column.

## Testing Standards

- SQLite is the sandbox/CI-speed verification target; it is **not** authoritative for anything PostgreSQL-specific (`FOR UPDATE SKIP LOCKED`, native enum DDL). Real PostgreSQL verification is the final word before any merge.
- Every new API endpoint needs explicit tests for: success path, cross-organization 404, not-found 404, and (where applicable) invalid-state-transition responses — no endpoint ships without this, following the standard set (and initially missed, then corrected in 5.1) by the profile endpoint.
- Every new worker handler needs an idempotency test proving a retried run doesn't duplicate its side effect (unique-constraint-plus-refetch pattern).
- Any handler touching tenant-supplied file paths needs an explicit cross-tenant-isolation test, not just a traversal-escape test — the two are different failure modes, both required (established by the B1 fix).
- A self-referential FK on a table (e.g. `task_runs.source_task_run_id → task_runs.id`, `RESTRICT`) needs the shared test-cleanup fixture to null out self-referencing columns before its bulk-delete pass — a plain per-table `DELETE` can otherwise violate `RESTRICT` depending on row order within that single statement (established by the Module 6 `_clean_tables` fix in `tests/conftest.py`).
- SQLite enforces neither PostgreSQL's 63-byte identifier length limit nor, independently, the `CHECK` constraint a migration-local native-enum column declares unless that migration explicitly sets `create_constraint=True` (this project's migrations never do — only the ORM model's own copy of the enum object does, and Alembic migrations never consult the ORM model). Both were confirmed empirically against a real Alembic-migrated SQLite database during Module 7, not assumed. Neither class of bug is catchable on SQLite alone — real-PostgreSQL verification (Phase 5) is what actually confirms identifier-length and enum-constraint correctness, not just migration-cycle symmetry.
- `tests/conftest.py`'s `_create_tables` fixture builds tables via `Base.metadata.create_all()`, not Alembic — against real PostgreSQL this requires the native enum *types* to already exist first (`create_type=False` on the ORM models means `create_all()` never creates them itself), so `alembic upgrade head` must be run once against the target Postgres database before the first test run against it. Re-running it on a later session is safe (a no-op once already at head) — documented in the README's "Running the same suite against real PostgreSQL" section, re-confirmed during Module 7's Phase 5 verification.
- Current baseline: 422 tests passing (363 baseline + 59 added in Module 8), full suite re-run required before every merge.

## Security Principles

- Tenant isolation is structural, not incidental: composite FKs, partial unique indexes scoped to active rows, and — as of 5.1 — per-tenant filesystem directories for any tenant-supplied file path. A path is never resolved against a shared root; it's resolved against that tenant's own root, with traversal-escape checks on top, not instead.
- Secrets never live in general-purpose metadata columns — `connection_metadata` is scanned against a secret-key denylist before it's ever persisted. Real credentials live only in `DataSourceCredential`, encrypted at rest (Fernet), accessed only through the `CredentialProvider` abstraction so the execution engine never knows or cares where secrets actually live.
- Concurrency safety under multiple workers is a fencing-token design (`lease_token`, regenerated per claim), not a trust-the-worker-id design — a worker whose lease was reclaimed can never corrupt state it no longer owns, even if it wakes up late.
- Every endpoint sits behind the same authentication dependency; there is no endpoint-specific auth mechanism to keep track of.

## Project Rules

Standing rules for all future work on this project, accumulated from explicit direction given throughout its development:

- Never rebuild, redesign, or restart a completed module without an explicit, specific reason.
- Never guess. If an assumption about existing behavior is uncertain, inspect the actual code before proposing or making a change.
- Never invent architecture that isn't already present — new modules extend existing patterns (handler registry, tenant-scoped FKs, bounded/sampled persistence) rather than introducing parallel ones.
- Never skip testing, and never treat SQLite-only verification as sufficient for a PostgreSQL-targeted change.
- No breaking changes to existing API contracts without an explicit, separate decision to do so.
- Preserve backward compatibility by default; additive changes are the norm.
- Recommend one production-ready approach with justification rather than presenting multiple alternative architectures, unless alternatives are explicitly requested. Note significant trade-offs briefly, then proceed with the recommendation.
- Wait for explicit approval before implementation begins on any new module, and before any merge to `main`.

## Known Limitations

- `SELECT ... FOR UPDATE SKIP LOCKED` claim-concurrency safety is PostgreSQL-only and cannot be exercised on SQLite — sandbox test results for claim-atomicity are indicative, not conclusive; real-Postgres verification is what actually confirms it.
- No in-process cancellation of a running worker handler — `WORKER_DEFAULT_TIMEOUT_SECONDS` governs lease-based reclaim by the reaper, not interruption of work already in progress. A pathologically slow handler run isn't stopped mid-flight, only eventually reclaimed.
- CSV is the only supported source type for real execution (`SourceType.POSTGRES`/`MYSQL`/`REST_API`/`S3`/`OTHER` all fail permanently under `SYNC`); connectors for other source types are unbuilt.
- CSV files are read entirely into memory, bounded by `CSV_MAX_FILE_SIZE_BYTES` (25 MB default) — no streaming path for larger files.
- Sandbox test/verification runs on Python 3.10; the production Docker image targets Python 3.13. These have not been cross-verified against each other in this project's history — worth confirming before a first real production deploy.
- No AI-assisted or ML-based data correction anywhere in the system yet — Module 6's cleaning engine is deterministic-rules-only by design, with AI-assisted correction explicitly deferred as a defined extension point, not built.
- Module 6 rollback is whole-run only (no cell-level selective undo), and rolled-back/rejected cleaning runs leave their output files on disk rather than deleting them — a storage retention policy is deferred, not solved.
- Module 7's `StandardizationConfig` (organization-configured default country/currency, date/boolean output display form, numeric locale) has no database persistence or API surface yet — only the two config tables specified in the approved design (`standardization_column_mappings`, `standardization_lookup_entries`) are implemented and wired. Every standardization run currently uses all-default config, so phone/state-province/postal-code standardization without a resolvable country column, the ambiguous `$` currency symbol, and locale-ambiguous numeric separators are conservatively left untouched rather than guessed.
- Module 7's casing rules are deliberately not locale-aware (no special-casing of "McDonald", "O'Brien", or lower-case connector words) and its postal-address/city rules do no structural parsing, canonical-name lookup, or USPS CASS-style certification — light-touch format normalization only, a named scope limit from the design, not an oversight.
- Module 7 rollback is whole-run only, same scope limit as Module 6, and likewise never deletes the standardized output file on rollback/rejection.
- Module 8 performs no physical merge, export, or deletion of duplicate records anywhere — approving a `MatchRun` is a pure status transition confirming the discovered groupings are correct, never an automatic action on the underlying data. Acting on those groupings (producing an actual deduplicated dataset) is a deliberately separate, unbuilt future capability, gated on the still-unbuilt `EXPORT` handler and its own future design/approval step.
- Module 8's canonical-record selection is fixed (lowest `row_index` / first-seen) and not organization-configurable — content-aware strategies ("most complete row," "most recently updated") are a named future extension point, not built.
- Module 8's blocking supports only a single highest-weight field as the blocking key; multi-field/composite blocking (which could catch more true duplicates that happen to disagree on the single highest-weight field) is a deferred accuracy improvement.
- Module 8 does no fuzzy, phonetic, or approximate matching of any kind (`comparison_type` is limited to `exact`/`normalized_exact`) — a hard, deliberate scope limit for explainability and reproducibility, not an oversight, and going further than the project's general "no AI/ML" rule by excluding classical approximate-matching algorithms too.
- Module 8's `match_rule_sets`/`match_rule_fields` configuration is immutable and versioned with no partial-update endpoint — changing even one field's weight requires creating an entirely new rule set (and re-running matching to see its effect); a future "clone and adjust" convenience endpoint is not precluded but isn't built.
- Module 8 matches exactly one `StandardizationRun` per `MatchRun` (no cross-run matching across multiple standardization runs/tasks/data sources within a single match run), and rollback is whole-run only, same scope limit as Modules 6/7.
