# Module 8 â€” Data Matching & Deduplication
## Design Specification (Design Only â€” No Code)

---

This spec was written against the actual current codebase (`main` at commit `d8c4f75`, tag `v0.7`, Alembic head `d4e5f6a7b8c9`) â€” every reuse claim below (handler shape, tenant FK convention, approval state machine, idempotency mechanism, bounded/capped persistence, enum DDL ownership, the identifier-length lesson from Module 7) is drawn from files read directly in this repository, not assumed.

## 1. Goals

- Detect duplicate and near-duplicate records within a single organization's **approved** Module 7 standardized output â€” matching is the fourth stage of the pipeline (profile â†’ clean â†’ standardize â†’ match), never applied to raw, unprofiled, uncleaned, or unstandardized data, and never applied to a standardized result a human hasn't yet approved.
- Support four escalating matching strategies over the same standardized dataset: exact duplicate detection (whole-row equality, no configuration required), normalized exact matching (a single configured column, compared after deterministic normalization), multi-column composite matching (several configured columns, each independently weighted), and, implicitly, no matching at all for columns/data an organization hasn't configured â€” never a forced guess at what "should" be compared.
- Group matched records into duplicate clusters, select one canonical record per cluster deterministically, and record every match decision â€” including every decision that did *not* produce a group â€” with its compared fields, normalized values, rule, field-level scores, total score, threshold, decision, confidence, reason, rule version, and timestamp.
- Require explicit human approval before a match result is treated as authoritative, with instant, non-destructive rollback â€” identical guarantee to Modules 6 and 7, extended to a domain (grouping, not transforming) that makes the "non-destructive" guarantee easier to keep, not harder: this module never rewrites, merges, or deletes any record anywhere.
- Support organization-specific matching configuration: which columns are match keys, how each is compared, how much each contributes to a match's total score, and where the duplicate/ambiguous thresholds sit â€” mirroring Module 7's organization-configuration precedent (`standardization_column_mappings`/`standardization_lookup_entries`) for a structurally different, more structured configuration shape (a versioned rule *set* containing multiple weighted field rules, not flat key/value rows).
- Never use AI, ML, or any non-deterministic or approximate-matching technique (no edit-distance, phonetic, or statistical similarity) anywhere in the matching path. Every comparison is a named, versioned, pure function; every score is a fixed, explainable weighted sum; every threshold is an explicit, organization-configured (or built-in default) number, never inferred.
- Reuse Module 4's execution engine, Module 5/6/7's tenant-scoped file I/O and approval-state-machine pattern, Module 3's validation helpers, and the existing API conventions without modification â€” Module 8 is a new handler and a new data surface on unchanged infrastructure, exactly as Module 7 was on Module 6's.

**The only integration seams into existing code:**
1. `HANDLER_REGISTRY[TaskType.MATCH]` â€” a new registry entry (see the required `TaskType` addition below).
2. `POST /tasks/{id}/runs`'s existing `TaskRunCreate.source_task_run_id` validation gains one more accepted `task_type` (`MATCH`), extending the exact branch Module 7 already extended from `(TRANSFORM,)` to `(TRANSFORM, STANDARDIZE)` â€” now `(TRANSFORM, STANDARDIZE, MATCH)`.

**Architectural decision â€” a new `TaskType.MATCH` enum value is required, not a reuse of an existing one.** Same reasoning Module 7 already established and re-verified directly against `app/worker/handlers/__init__.py`'s current docstring: `SYNC` means profiling, `TRANSFORM` means cleaning, `STANDARDIZE` means standardization, `EXPORT` is explicitly reserved for a future export handler (unbuilt, `NoOpHandler`), and `OTHER` is the generic no-op fallback used by nothing in particular. None of these can be reused without overloading an existing type's meaning. Recommended and proceeding with a new enum value, extended via `ALTER TYPE ... ADD VALUE` exactly as Module 7's migration already did (Section 3) â€” the second migration of this kind in the project, not the first, so the mechanics are now a repeated, proven pattern rather than a novel one.

**Architectural decision â€” Module 8 produces no output file, unlike Modules 6 and 7.** Cleaning and standardization each transform every cell of a dataset, so their natural artifact is a rewritten CSV plus a per-cell change log. Matching does not transform any value â€” it *groups* existing standardized records and proposes a canonical member per group. Writing a "deduplicated" CSV would mean physically merging or dropping rows, which is exactly the destructive, automatic action the acceptance criteria forbid ("no automatic destructive deletion," "never overwrite original, cleaned, or standardized data"). Module 8's deliverable is therefore the audit data itself â€” `MatchRun`, `MatchGroup`, `MatchDecision` rows â€” not a file. Producing an actual deduplicated export remains a deliberately separate, unbuilt, future capability (the already-reserved `TaskType.EXPORT`, per Module 6/7's own Future Roadmap entries â€” see Section 16), gated on its own future approval step, not something this module does implicitly. This is not new architecture â€” it is *less* architecture than Modules 6/7 needed, chosen because the domain genuinely doesn't require a file artifact, not invented to fill a gap.

## 3. Database Changes

**One enum addition.** `task_type_enum` gains a fifth-used value, `match` (`TaskType.MATCH`). Mechanically identical to Module 7's `standardize` addition (Section 3 of `docs/module-7-data-standardization-engine-design.md`), now a proven, repeated pattern rather than a first attempt:

- **PostgreSQL:** `ALTER TYPE task_type_enum ADD VALUE IF NOT EXISTS 'match'`, issued inside `op.get_context().autocommit_block()` â€” required because the newly added label cannot be used within the same transaction it was added in (PostgreSQL 12+, this project targets 16).
- **SQLite:** no change needed, re-confirmed against this project's own empirical finding from Module 7: `tasks.task_type` has no `CHECK` constraint on a real Alembic-migrated SQLite database (migration-local enum objects in this project never set `create_constraint=True`), and SQLite doesn't enforce declared `VARCHAR` lengths either.
- **Downgrade:** same documented, standard limitation as Module 7 â€” PostgreSQL has no `DROP VALUE`; the label is left in place on downgrade, which does not affect a genuine `downgrade base â†’ upgrade head` cycle's correctness in a fresh verification database.

**A lesson applied proactively this time, not discovered again:** Module 7's real-PostgreSQL verification caught an auto-derived foreign-key name exceeding PostgreSQL's 63-byte (`NAMEDATALEN`) identifier limit, because that project's `fk_{table}_{column}_{ref_table}` naming convention doesn't scale to longer table names. Every constraint name below uses the project's existing short `_org_`-abbreviated form from the start (e.g. `fk_match_runs_org_id`, not `fk_match_runs_organization_id_organizations`) â€” every name was measured against the 63-byte limit before being written here; the longest is 34 characters (`fk_match_decisions_org_match_group`), comfortably under the limit with no per-migration risk of repeating Module 7's finding.

**Five new tables** â€” three audit tables mirroring `standardization_runs`/`standardization_changes`' shape and constraint discipline, plus two new organization-configuration tables mirroring `standardization_column_mappings`/`standardization_lookup_entries`'s discipline for a more structured (parent + child) configuration shape:

### `match_runs` â€” one row per `MATCH` `TaskRun`

Same one-row-per-run idempotency pattern as every prior run-audit table (`UniqueConstraint(task_run_id)`), `UniqueConstraint(organization_id, id)` present from the start (the Module 6 lesson, already applied without incident in Module 7, applied again here). Composite tenant FKs to `task_runs` (`task_run_id`, CASCADE), `tasks` (RESTRICT), `data_sources` (RESTRICT), `task_runs` again (`source_task_run_id`, RESTRICT â€” the `STANDARDIZE` run being matched), and `match_rule_sets` (`rule_set_id`, RESTRICT, **nullable** â€” see below).

Columns: `id`, `organization_id`, `task_run_id`, `task_id`, `data_source_id`, `source_task_run_id`, `rule_set_id` (nullable â€” `NULL` means no organization configuration existed at run time, so only the always-available exact-duplicate pass ran; never a forced default rule set), `rule_set_version` (nullable `Integer`, denormalized copy of the rule set's `version` at run time, so this run's provenance is fixed even if the organization later creates a newer rule-set version â€” same "denormalize the version that was actually in effect" precedent `StandardizationChange.rule_version` already established), `row_count`, `total_comparisons_count` (the true number of pairwise comparisons actually performed, post-blocking â€” see Section 6, never the theoretical `nÂ²`), `duplicate_group_count`, `duplicate_pairs_count`, `ambiguous_pairs_count` (both **true** totals, accurate even when persisted `MatchDecision` rows are capped â€” same bounded-but-never-silent contract as `CleaningRun.total_changes_count`/`StandardizationRun.total_changes_count`), `skipped_block_count` (blocks skipped for exceeding `MATCH_MAX_BLOCK_SIZE`, Section 11 â€” surfaced, never silent), `decisions_by_rule` (JSON, mirrors `changes_by_rule`), `confidence_score` (minimum confidence across every `duplicate`-decision that fed a group, `1.0` if zero groups â€” identical semantics to Module 6/7's confidence aggregation), `match_engine_version` (`String(20)`, NOT NULL), `status` (`String(20)`, default `pending_review`, `CHECK IN ('pending_review','approved','rejected','rolled_back')` â€” plain string, matching `cleaning_runs.status`/`standardization_runs.status`'s precedent, not a native enum, for the same "small internal worker-owned state field" reason), `approved_by`/`approved_at`, `rejected_by`/`rejected_at`, `rolled_back_by`/`rolled_back_at`, `created_at`.

No `output_file_path`/`output_sha256` columns â€” there is no output file (Section 2).

### `match_groups` â€” one row per duplicate cluster

Composite tenant FK to `match_runs` (CASCADE). `UniqueConstraint(organization_id, id)` present from the start, since `match_decisions.match_group_id` needs to target it as a composite FK.

Columns: `id`, `organization_id`, `match_run_id`, `canonical_row_index` (`Integer` â€” the `row_index`, within this run's loaded standardized CSV, of the deterministically-selected canonical record; see Section 8), `record_count` (`Integer`, `CHECK >= 2` â€” a group is never created for fewer than two records), `confidence_score` (`Float`, minimum confidence across this group's member `duplicate` decisions), `created_at`.

### `match_decisions` â€” one row per pairwise comparison that reached at least the review threshold

Composite tenant FK to `match_runs` (CASCADE) and, when set, to `match_groups` (CASCADE, **nullable** â€” `NULL` for `ambiguous` decisions, which are never grouped; see Section 7). This is the granular, potentially large audit table â€” capped per run at `MATCH_MAX_PERSISTED_DECISIONS` (Section 11), same bounded pattern as `CleaningChange`/`StandardizationChange`.

Columns:
- `id`, `organization_id`, `match_run_id`, `match_group_id` (nullable)
- `record_a_row_index`, `record_b_row_index` (`Integer` â€” the compared pair's `row_index` values within this run's loaded standardized CSV, the same `row_index` convention `CleaningChange`/`StandardizationChange` already use for "which record"; `CHECK (record_a_row_index < record_b_row_index)` enforces one canonical, non-redundant storage order for every pair â€” a pair is never stored twice in both orders)
- `rule_name` (`String(50)` â€” one of a fixed set, Section 7)
- `field_comparisons` (`JSON`, NOT NULL â€” one structured column consolidating the "compared fields," "normalized compared values," and "field-level scores" required per the acceptance criteria, rather than three separate columns or a sixth child table: `{"<column_name>": {"value_a": "<normalized>", "value_b": "<normalized>", "matched": true|false, "weight": <float>, "contribution": <float>}, ...}`. Every required item is present and independently addressable by key; consolidated because it is one comparison's internal detail, not a separately query-needed entity â€” the same rationale `changes_by_rule` already uses for aggregate data in this project, applied here to per-decision structured detail instead of a run-level aggregate.)
- `total_score` (`Float`, `CHECK 0 <= total_score <= 1` â€” the weight-normalized sum of `field_comparisons[*].contribution`)
- `threshold_used` (`Float`, `CHECK 0 <= threshold_used <= 1` â€” the `duplicate_threshold` in effect for this comparison, denormalized onto the decision itself so it stays self-documenting even if the organization's rule set changes later)
- `decision` (`String(20)`, `CHECK IN ('duplicate', 'ambiguous')` â€” a `not_duplicate` outcome is never persisted at all; see Section 7 for why this is a deliberate, bounded-persistence decision, not an omission)
- `confidence_score` (`Float`, `CHECK 0 <= confidence_score <= 1` â€” equal to `total_score` for this initial deterministic design; see Section 7 for why a separate, independent confidence metric isn't introduced)
- `reason` (`Text`, NOT NULL â€” a human-readable explanation, e.g. `"3 of 3 configured fields matched (score 0.92 >= threshold 0.85)"` or, for an ambiguous pair, `"2 of 3 configured fields matched (score 0.61 is between the review threshold 0.50 and the duplicate threshold 0.85) -- flagged for manual review, not grouped"`)
- `rule_version` (`String(20)`, NOT NULL â€” `MATCH_ENGINE_VERSION`, the same "code engine version, not the org's rule-set version" meaning `StandardizationChange.rule_version` already has; the org rule-set version is separately available via the parent `MatchRun.rule_set_version`)
- `created_at` (the required timestamp)

### `match_rule_sets` â€” organization-configured, versioned matching configuration

New relative to Module 7, required by the "organization-specific matching rules" and "configurable match keys" acceptance criteria, but following the same tenant-scoped, soft-deletable, partial-unique-index-on-the-active-scope convention `standardization_column_mappings` already established for a nullable-scope column. Rule sets are **immutable once created** â€” changing an organization's matching configuration means creating a new, higher-`version` row and deactivating the old one (`is_active = false`), never editing rows in place; this mirrors the project's existing "never overwrite, version instead" convention (`CLEANING_ENGINE_VERSION`/`STANDARDIZATION_ENGINE_VERSION` bump-on-change) applied to organization-supplied configuration instead of built-in code.

Columns: `id`, `organization_id`, `data_source_id` (nullable â€” `NULL` applies to every data source in this organization, same meaning as `standardization_column_mappings.data_source_id`; composite tenant FK when set, RESTRICT), `version` (`Integer`, `CHECK >= 1`, server-computed at creation time as "one more than this organization's current count for this scope," never client-supplied), `duplicate_threshold` (`Float`), `review_threshold` (`Float`), `CHECK (0 <= review_threshold AND review_threshold < duplicate_threshold AND duplicate_threshold <= 1)` (a well-defined, non-overlapping ambiguous zone is enforced at the database layer, not just the API layer), `is_active` (`Boolean`, default `true`), `created_by`, `created_at`.

Two partial unique indexes (the same nullable-column pattern `standardization_column_mappings` uses, keyed on scope rather than a name, since the invariant here is "at most one active rule set per scope," not "at most one active mapping per column"): `ix_match_rule_sets_scoped_active` on `(organization_id, data_source_id)` where `is_active AND data_source_id IS NOT NULL`; `ix_match_rule_sets_orgwide_active` on `(organization_id)` where `is_active AND data_source_id IS NULL`.

### `match_rule_fields` â€” the weighted field list for one rule set

Created together with its parent `match_rule_sets` row in a single API call (Section 5) and never edited or added to afterward â€” a rule set's field list is part of its immutable, versioned definition, so there is no separate update endpoint for this table, only creation (as part of the parent) and read.

Columns: `id`, `organization_id`, `rule_set_id` (composite tenant FK to `match_rule_sets`, CASCADE â€” deleting a rule set is never done in practice, since rule sets are soft-deactivated, but the FK is CASCADE for referential correctness if a row is ever hard-deleted in a future administrative cleanup), `column_name` (`String(255)`, matched case-insensitively against the standardized CSV's headers, same convention `standardization_column_mappings.column_name` uses), `comparison_type` (`String(20)`, `CHECK IN ('exact', 'normalized_exact')` â€” validated against this same fixed set at the Pydantic layer too, matching `standardization_column_mappings.field_type`'s "small closed set â†’ plain string, not a native enum" precedent), `weight` (`Float`, `CHECK > 0`), `created_at`. `UniqueConstraint(rule_set_id, column_name)` â€” the same column can't be configured twice within one rule set (which would make weight-summing ambiguous).

**Zero changes** to any Module 1â€“7 table. `task_runs.source_task_run_id` is reused as-is with no new column, exactly as Module 7 reused it for pointing at a `TRANSFORM` run instead of a `SYNC` run.

Migration: one new file, `down_revision` pointing at the current head (`d4e5f6a7b8c9`) â€” a single new head, verified before merge exactly as every prior module's migration was.
## 4. Worker Flow

No changes to `app/worker/engine.py`, `reaper.py`, `metrics.py`, or `runner.py` â€” a match run is claimed, leased, heartbeated, retried, and completed by the exact same code path a profiling, cleaning, or standardization run uses today. `HANDLER_REGISTRY[TaskType.MATCH]` is a one-entry registry change, not new dispatch logic.

`MatchHandler.execute` mirrors `StandardizationHandler.execute`'s five-stage shape:

1. **Resolve inputs.** Validate the data source is an active `CSV_UPLOAD` (same check every prior CSV-consuming handler already does). Validate `context.task_run.source_task_run_id` is set. Look up the `StandardizationRun` for that `source_task_run_id`, scoped by org â€” missing is a permanent failure ("matching requires a completed standardization run for source_task_run_id"). **Require `StandardizationRun.status == "approved"`** â€” a `pending_review`, `rejected`, or `rolled_back` standardization result is not yet authoritative and must not feed matching, mirroring Module 7's identical gate on `CleaningRun.status` word-for-word; permanent failure, not retryable, since the condition won't resolve without a human decision. Check for an existing `MatchRun` by `task_run_id` first (idempotency short-circuit, same pattern as every prior handler).
2. **Load.** Read `StandardizationRun.output_file_path` â€” the tenant-scoped Module 7 output, not the Module 5 or Module 6 outputs and not the original source â€” via the existing, unchanged `load_csv`/`CsvLimits`. No new size/row/column limit logic.
3. **Resolve configuration.** Look up an active `match_rule_sets` row for this org, preferring a data-source-specific row over an org-wide (`data_source_id IS NULL`) one, exactly the same precedence `StandardizationHandler._load_column_overrides` already applies. If neither exists, `rule_set` is `None` and only the always-available exact-duplicate pass runs (Section 6) â€” a safe, valid, and complete outcome, not a permanent failure and not a forced guess at default match keys (unlike field-type classification, there is no safe universal heuristic for "which columns identify a duplicate record" â€” that is inherently organization- and dataset-specific).
4. **Match.** Run the deterministic matching engine (Section 6) over the loaded rows, producing duplicate groups, a bounded decision list, and aggregate counts.
5. **Persist.** Write one `MatchRun` row plus bounded `MatchGroup` and `MatchDecision` rows in one transaction, using the identical unique-constraint-plus-refetch idempotency pattern every prior handler uses (including the same `id=uuid.uuid4()`-assigned-explicitly-before-flush discipline, applied from the start rather than discovered again). Set initial status `pending_review`.

## 5. API Endpoints

All additive, all under the existing `tasks` router, following Modules 6/7's precedent of extending the same file rather than introducing a new one; no existing endpoint's signature, response model, or behavior changes.

- `POST /tasks/{id}/runs` â€” `TaskRunCreate.source_task_run_id` becomes required for `MATCH` tasks too (the same `400`/`404` validation branch already extended once from `(TRANSFORM,)` to `(TRANSFORM, STANDARDIZE)`, now extended to `(TRANSFORM, STANDARDIZE, MATCH)`), and remains rejected for `SYNC`/`EXPORT`/`OTHER`. The API-layer check confirms only that the referenced run exists in the same org; the deeper "must be an approved `StandardizationRun`" check stays in `MatchHandler`, the same validation-layering precedent Module 7 already followed for `TRANSFORM`/`STANDARDIZE`.
- `GET /tasks/{id}/runs/{run_id}/matching` â€” the run summary (counts, confidence, approval status). `404` if the run isn't visible to the caller's org or no result exists yet. Mirrors `GET .../standardization` exactly, minus the output-file fields Module 8 doesn't have.
- `GET /tasks/{id}/runs/{run_id}/matching/groups` â€” paginated list of `MatchGroup` rows for the run, in `canonical_row_index` order.
- `GET /tasks/{id}/runs/{run_id}/matching/decisions` â€” paginated list of `MatchDecision` rows for the run, optionally filtered by `?decision=duplicate` or `?decision=ambiguous` (the latter is the organization's reviewable "ambiguous match" queue â€” a dedicated, queryable surface for exactly the records the acceptance criteria say must never be silently merged) and/or `?match_group_id=...` to see every decision that contributed to one specific group.
- `POST /tasks/{id}/runs/{run_id}/matching/approve` / `.../reject` / `.../rollback` â€” the identical three-endpoint approval state machine Modules 6/7 already implement (`pending_review â†’ approved`/`rejected`, `approved â†’ rolled_back`, `409` on any other starting status), reused pattern-for-pattern with the model swapped. Approving a `MatchRun` performs no destructive action of any kind â€” it is a pure status transition confirming the organization accepts these groupings as correct, exactly as read above in Section 2's architectural decision; nothing is deleted or merged at approval time, by design, not as a deferred safety net.
- **Organization configuration CRUD** (new surface, following Module 7's precedent of a new-but-consistent path prefix under the same router): `POST`/`GET` on `/tasks/matching/rule-sets` â€” `POST` creates a rule set and its full field list in one call (the field list is part of the request body, e.g. `{"data_source_id": null, "duplicate_threshold": 0.85, "review_threshold": 0.5, "fields": [{"column_name": "email", "comparison_type": "normalized_exact", "weight": 1.0}, ...]}`), computing `version` server-side and deactivating any prior active rule set for the same scope in the same transaction (soft "supersede," not a delete â€” the old version remains readable, satisfying the same non-destructive-configuration-history principle as everywhere else in this project). `GET` lists rule sets (with their nested field lists) for the organization, `?include_inactive=true` to see prior versions. No `PATCH`/`PUT` (rule sets are immutable) and no `DELETE` (deactivation happens automatically when a new version supersedes it, and a rule set that produced historical `MatchRun` rows should never disappear from view â€” every `MatchRun.rule_set_id`/`rule_set_version` must remain resolvable).

## 6. Matching and Deduplication Engine

`app/matching/`, mirroring `app/standardization/`'s package shape (`types.py`, `engine.py`, `rules/` â€” here `blocking.py`, `comparison.py`, `clustering.py`, `canonical.py` rather than one module per field-type family, since Module 8's rule surface is organized around comparison *strategy*, not field type). Every function pure, no I/O, no randomness, no wall-clock dependence beyond what's passed in explicitly. `MATCH_ENGINE_VERSION` (initial `"1.0"`) is bumped whenever any function's *output* could change for existing input â€” the same contract `CLEANING_ENGINE_VERSION`/`STANDARDIZATION_ENGINE_VERSION` already have.

**Stage 1 â€” Exact duplicate detection (always runs, needs no configuration).** Every row is reduced to a single deterministic key: the tuple of every column's value, in header order, joined with a fixed separator. Rows sharing an identical key are grouped immediately â€” no comparison, no scoring, `rule_name = "exact_row_match"`, `total_score = confidence_score = 1.0`. This is the cheapest, safest, always-available detection method (an O(row_count) dictionary pass), and it runs whether or not the organization has configured anything.

**Stage 2 â€” Composite/normalized matching (runs only if an active `MatchRuleSet` was resolved).** Rows already grouped by Stage 1 are represented by exactly one member (their lowest `row_index`, the same deterministic "first-seen wins" tie-break this project already uses for header-pattern classification and country-name-variant resolution) â€” comparing every original row again would be redundant work against rows already known to be identical. Every ungrouped row, plus one representative per Stage-1 group, proceeds to Stage 2.

- **Blocking (bounds the comparison space).** The single `MatchRuleField` with the highest `weight` for the resolved rule set (deterministic tie-break: lowest `id`, i.e. creation order, if weights tie) is the *blocking key*. Every candidate row's value in that column is normalized (trim + casefold, reusing `app.cleaning.rules.trim_whitespace` the same defensive-second-pass way Module 7 already does) and used to bucket rows; only rows sharing an identical, **non-empty** blocking-key value are ever compared to each other. Rows with a blank blocking-key value are never matched against anything by Stage 2 (matching on emptiness would be a guaranteed false-positive multiplier â€” the same "never guess past an unresolved ambiguity" principle Module 7 applies to unresolvable countries/currencies). This bounds comparison work to the sum of each block's `sizeÂ²` rather than the full dataset's `nÂ²`; only a single-field blocking key is supported in this release (Section 16).
- **Per-pair comparison.** For every pair of rows sharing a block, every configured `MatchRuleField` is evaluated: `comparison_type = "exact"` compares the raw (already-standardized) values byte-for-byte; `comparison_type = "normalized_exact"` compares them after the same trim+casefold normalization used for blocking. Each field contributes `weight` to the numerator if it matches, `0` if it doesn't; `total_score = (sum of matched fields' weights) / (sum of every configured field's weight)`, always in `[0, 1]`, regardless of how many fields happen to match â€” a rule set with exactly one configured field is exactly "normalized exact matching" (or "exact matching," depending on that field's `comparison_type`); a rule set with several is "multi-column composite matching." Both are the same engine code path, differing only in configuration, per the acceptance criteria's expectation that these are related, not separately-implemented, capabilities.
- **Threshold classification.** `total_score >= duplicate_threshold` â†’ `decision = "duplicate"`; `review_threshold <= total_score < duplicate_threshold` â†’ `decision = "ambiguous"`; `total_score < review_threshold` â†’ no decision is persisted at all (Section 3 explains why: persisting every non-match pair would be unbounded and useless, and "no decision" is itself the correct, safe, explainable outcome for a clear non-match â€” the aggregate `total_comparisons_count` on `MatchRun` still reflects the true number of pairs evaluated, so this is bounded persistence, not silent data loss).
- **Block-size bound.** A block whose size exceeds `MATCH_MAX_BLOCK_SIZE` is skipped entirely â€” no pairwise comparisons are performed within it. This is a deliberate, surfaced bound (`MatchRun.skipped_block_count` records how many blocks this happened to), never a silent truncation of results; rows in a skipped block may still have been caught by Stage 1's exact-match pass.

**Stage 3 â€” Clustering.** Every `duplicate`-decision pair is unioned via a standard, deterministic union-find (disjoint-set) structure into connected components â€” so if Aâ€“B and Bâ€“C are both `duplicate` decisions, A, B, and C form one group even though Aâ€“C may never have been directly compared (different block, or scored just under threshold on a direct comparison that didn't happen). `ambiguous` decisions are never unioned and never create or extend a group â€” they are recorded with `match_group_id = NULL` and surfaced only via the decisions endpoint's `ambiguous` filter (Section 5), which is the structural mechanism, not a policy note, behind "ambiguous matches are never silently merged."

## 7. Match Rules and Scoring

**Fixed rule-name registry**, mirroring `RULE_CONFIDENCE`/`RULE_REASONS`'s shape from Modules 6/7 but scoring-derived rather than fixed-per-rule (Section 3 explains why `confidence_score` isn't an independent metric here): `MATCH_RULE_EXACT_ROW` (Stage 1), `MATCH_RULE_NORMALIZED_EXACT` (Stage 2, single configured field), `MATCH_RULE_COMPOSITE_WEIGHTED` (Stage 2, multiple configured fields) â€” resolved from the count of configured `MatchRuleField` rows at comparison time, not stored per-field.

**Confidence is defined as equal to `total_score`.** Modules 6/7's per-rule `RULE_CONFIDENCE` table encodes "how much do we trust this *class* of transformation" â€” a property of the rule, not the specific input. Module 8's matching confidence is fundamentally a property of *how well the specific compared values agreed*, which `total_score` already measures precisely; introducing a second, independent confidence number would either duplicate `total_score` or require an unjustified new heuristic. Stage 1 (`exact_row_match`) is the one case with a fixed confidence (`1.0`), since whole-row byte-identity leaves nothing further to score.

**Threshold configuration is entirely organization-supplied**, with one exception: if no active `MatchRuleSet` exists for an organization/data source, Stage 2 does not run at all (Section 6) â€” there is no built-in default `duplicate_threshold`/`review_threshold` to fall back to, because a default threshold implies a default idea of "what counts as similar enough," which (unlike Module 7's header-name classification heuristic) has no safe, universal answer across arbitrary datasets. An organization that wants composite/normalized matching must configure it; Stage 1 (exact duplicates) always runs regardless and requires no configuration or threshold at all.

**No fuzzy, phonetic, or approximate comparison of any kind** â€” `comparison_type` is limited to `exact` and `normalized_exact` in this release (Section 16); this is the classical-algorithm equivalent of Modules 6/7's "no AI/ML" stance, verified the same way (grepping the new package for any similarity-scoring, edit-distance, or probabilistic library).
## 8. Canonical-Record Selection

Within every duplicate `MatchGroup`, exactly one member is selected as the canonical record, recorded as `MatchGroup.canonical_row_index`. The sole rule in this release: **the lowest `row_index` in the group** â€” the first-seen record, a fixed, deterministic, content-independent tie-break, consistent with every other "first match wins" convention already established in this project (header-pattern classification, built-in country/state lookup resolution). Canonical selection is deliberately not organization-configurable in this release (Section 16) â€” "most complete row," "most recently updated," or any other content-aware strategy would require additional configuration surface and additional judgment calls (e.g., what does "most complete" mean for a partially-blank standardized row?) beyond what's been reviewed and approved here; a clearly-named future extension point, not a silent gap. Selecting a canonical record never modifies, merges, or deletes any group member â€” it is purely a labeled recommendation surfaced for a human (or a future, separate export/merge step) to act on.

## 9. Audit Model

Two append-only, per-run audit tables plus one grouping table, directly extending Module 6/7's audit-model precedent (Section 11 of the Module 6 design, Section 7 of the Module 7 design) to a domain where the audited unit is a *comparison* or a *cluster*, not a per-cell transformation:

- `MatchRun` â€” one per `MATCH` `TaskRun`, same one-row-per-run idempotency pattern as `CleaningRun`/`StandardizationRun`.
- `MatchGroup` â€” one per duplicate cluster: which records, which one is canonical, the cluster's confidence.
- `MatchDecision` â€” one per pairwise comparison that reached at least the review threshold: the exact six-plus-required-fields the acceptance criteria specify (source record identifiers, duplicate group identifier, compared fields, normalized compared values, rule applied, field-level scores, total score, threshold used, decision, confidence, reason, rule version, timestamp) â€” every one of these is a distinct, independently addressable piece of `MatchDecision`, satisfying the acceptance criteria field-for-field, not by approximation.

All three follow the same tenant-scoping convention every table since Module 3 uses: `organization_id` on every row, composite FK to the parent scoped by organization, no new isolation model introduced. None are ever updated or deleted after creation, including after rollback â€” the record of what was compared, scored, and decided persists regardless of what happens to the parent `MatchRun`'s approval state, identical to `CleaningChange`/`StandardizationChange`'s guarantee.

## 10. Approval and Rollback Strategy

Identical mechanism and guarantees to Modules 6/7's (Section 12 of the Module 6 design, Section 8 of the Module 7 design), reused pattern-for-pattern â€” and, as Section 2 notes, structurally *simpler* to guarantee here, since Module 8 never writes an output file or performs any destructive action at any point in its lifecycle, approved or not.

- Approval state machine: `pending_review â†’ approved`/`rejected`; only an `approved` run can later be `rolled_back`; every transition records the acting user and timestamp; `409 Conflict` on any other starting status.
- **Approving** a `MatchRun` is a pure status transition. It does not merge, delete, or modify any record, any file, or any other database row â€” it marks the organization's confirmation that the discovered groupings are correct, available for a future, separate, explicitly-approved export/merge step to act on (Section 16), never performed automatically or implicitly by this module.
- **Rolling back** an approved `MatchRun` sets `status = rolled_back` and records who/when; every `MatchGroup` and `MatchDecision` row is untouched, so a rolled-back run's full history remains inspectable â€” since nothing destructive happened at approval time in the first place, rollback here is a strictly simpler, lower-risk operation than Module 6/7's (where rollback matters because an approved cleaning/standardization result superseded prior data for downstream consumption; here there is no superseded artifact to worry about un-superseding).
- Rollback is whole-run only in this release, same scope limit as Modules 6/7.
## 11. Performance Considerations

- Reuses Module 5's existing row/column/cell-size bounds as-is (via the same `CsvLimits`/`load_csv`) â€” no new ingestion ceiling to define.
- **Blocking is the primary compute bound**, not an optimization applied on top of an otherwise-unbounded algorithm: Stage 2 never compares two rows unless they share a non-empty blocking-key value, turning the worst-case `O(nÂ²)` full cross-product into the sum of each block's `sizeÂ²` â€” for evenly distributed data this is close to linear in practice, and the explicit `MATCH_MAX_BLOCK_SIZE` bound (a new setting, default to be chosen conservatively, e.g. `1,000`) caps the worst case for any single skewed block, with the skip surfaced via `MatchRun.skipped_block_count` rather than silently degrading or hanging.
- `MatchDecision` persistence is explicitly capped (a new setting, `MATCH_MAX_PERSISTED_DECISIONS`, default `10,000`, mirroring `CLEANING_MAX_PERSISTED_CHANGES`/`STANDARDIZATION_MAX_PERSISTED_CHANGES`) with always-accurate aggregate counts (`total_comparisons_count`, `duplicate_pairs_count`, `ambiguous_pairs_count`, `decisions_by_rule`) on `MatchRun` â€” the same bounded-but-never-silent pattern established twice already, applied a third time.
- `MatchGroup` rows are not separately capped â€” a dataset's group count is structurally bounded by its row count (at most `row_count / 2` groups), several orders of magnitude smaller than the potential decision count, so no additional ceiling is needed beyond what blocking and the decision cap already provide.
- The organization's `MatchRuleSet`/`MatchRuleField` configuration is read once per run (a handful of rows), not re-queried per comparison â€” the same "load configuration once before the row loop" discipline `StandardizationHandler._load_lookup_tables` already established.
- Inherits Module 4's existing, already-documented property that a running handler cannot be preemptively cancelled mid-execution â€” not a regression Module 8 introduces.

## 12. Security and Tenant Isolation

- **Tenant isolation** is enforced the same way every prior module establishes it: matching only ever loads one organization's standardized output (already tenant-scoped at the `StandardizationRun`/`DataSource` level); every new table follows the existing composite-FK tenant-scoping pattern; `match_rule_sets`/`match_rule_fields` are both `organization_id`-scoped and read only within that scope, so one organization's matching configuration can never affect another's data. Because matching never reads or writes anything outside the single `StandardizationRun` resolved in Stage 1 of the handler (Section 4), and that run was itself looked up scoped to `context.task_run.organization_id`, **cross-organization comparison is structurally impossible, not merely policy-forbidden** â€” there is no code path through which a row from organization A's standardized CSV could ever be loaded into the same in-memory comparison pass as organization B's.
- **Auth**: every new endpoint sits behind the same `get_current_active_user` dependency every existing endpoint uses. No new authentication or authorization mechanism.
- **No new credential surface.** `MatchHandler` receives the same `ExecutionContext` as every handler; it does not touch `CredentialProvider` or `DataSourceCredential` at all, matching every prior CSV-consuming handler's existing scope.
- **Configuration input validation.** `match_rule_fields.comparison_type` is validated against the fixed `{"exact", "normalized_exact"}` set at the Pydantic layer (a `422`, never a silent no-op on an unrecognized value); `match_rule_sets.duplicate_threshold`/`review_threshold` ordering (`0 <= review_threshold < duplicate_threshold <= 1`) is validated at both the Pydantic layer and the database `CHECK` constraint, so a malformed threshold configuration can never reach the engine; `column_name` passes through `app.core.validation.normalize_name` the same as every other user-supplied name field in this project.
- **File safety.** `MatchHandler` performs no writes of any kind beyond the database rows described in Section 3 â€” there is no output file, so there is no new file-write surface to secure at all (a smaller attack surface than Modules 6/7, not a gap).
- **No AI, no ML, no approximate matching, no external network calls.** Every comparison is a pure function (`exact` byte-equality or `normalized_exact` trim+casefold-equality) or a fixed weighted sum; verified in the production review by grepping the new package for any HTTP client, model-loading, similarity/edit-distance library, or non-deterministic (`random`, wall-clock-dependent) code path â€” the same verification method Module 7 already used, applied again.

## 16. Explicit Scope Limits and Known Limitations

- **No physical merge, export, or deletion of duplicate records anywhere in this module.** Acting on a `MatchRun`'s approved groupings â€” producing an actual deduplicated dataset â€” is a deliberately separate, unbuilt future capability (the already-reserved `TaskType.EXPORT`), gated on its own future design and approval, not something Module 8 does automatically or implicitly at any point, including after approval.
- **Canonical-record selection is fixed** (lowest `row_index` / first-seen) and not organization-configurable in this release. Content-aware strategies ("most complete row," "most recently updated") are a named future extension point, not built.
- **Blocking supports only a single highest-weight field as the blocking key.** Multi-field/composite blocking keys (which could catch more true duplicates that happen to disagree on the single highest-weight field) are a deferred optimization/accuracy improvement, not present in this release.
- **No fuzzy, phonetic, or approximate matching of any kind** â€” no edit-distance (Levenshtein/Jaro-Winkler), no phonetic (Soundex/Metaphone), no statistical or ML-based similarity. `comparison_type` is limited to `exact` and `normalized_exact`. This is a hard scope limit for this release, matching the "no AI/ML" requirement but going further to exclude classical approximate-matching algorithms too â€” a deliberate choice to keep every comparison perfectly explainable and reproducible, not an oversight.
- **A block exceeding `MATCH_MAX_BLOCK_SIZE` is skipped entirely, not sampled or truncated.** Rows in a skipped block receive no Stage-2 comparisons at all (though Stage 1's exact-match pass still applies to them); this is surfaced via `MatchRun.skipped_block_count`, never silent, but it does mean an organization with a very large number of rows sharing one blocking-key value will see those rows go uncompared by composite matching until they configure a more selective blocking field or a future release adds multi-field blocking.
- **`MatchRuleSet`/`MatchRuleField` configuration is immutable and versioned, with no partial-update endpoint.** Changing even one field's weight requires creating an entirely new rule set (and re-running matching to see its effect) â€” simpler and safer to reason about than in-place mutation of a configuration that past `MatchRun`s already reference, but less convenient for iterative threshold tuning; a future UI/API convenience (e.g., "clone and adjust") is not precluded but isn't built here.
- **Matching operates on exactly one `StandardizationRun` per `MatchRun`**, the same one-parent-per-run shape Modules 6/7 already use â€” there is no cross-run matching across multiple different standardization runs, tasks, or data sources within a single `MatchRun` in this release.
- **No cell-level or partial rollback** â€” rollback is whole-run only, same scope limit as Modules 6/7.
